# 시설 관리 기준 스케줄: 현재 테이블 vs 별도 운영 추천

시설(돈사·시설)에 대한 관리 기준이 **주 몇 회, 월별, 계절별** 등 여러 조건으로 스케줄화될 때,  
**현재 `schedule_items` 테이블로 통합할지**, **별도 구조로 운영할지**에 대한 추천입니다.

---

## 1. 현재 구조 요약

| 항목 | 내용 |
|------|------|
| 테이블 | `schedule_items` (구분=돼지/시설, 대상장소, 기준, 날짜(일수), 작업유형, 작업내용) |
| 기준 | `schedule_basis_types` (전입일, 출산일, 매일, 주 1회 등) — 구분(pig/facility) 지원 |
| 날짜 | `dayMin`, `dayMax` — **일수** (돼지: 전입일+0, +7 … 에 최적화) |

돼지 일정은 **“기준일 + N일”** 구조에 잘 맞고, 시설은 이미 `targetType: facility`로 같은 테이블에 들어갈 수 있습니다.

---

## 2. 시설 관리 조건의 특성

| 조건 예시 | 의미 | 현재 dayMin/dayMax와의 관계 |
|-----------|------|-----------------------------|
| 매일 | 매일 수행 | dayMin=0, dayMax=0 또는 null 로 표현 가능 |
| 주 1회 | 주 1회 (요일 무관 또는 특정 요일) | “1회”는 표현 가능, **요일**은 별도 필드 없음 |
| 주 2회 | 주 2회 (예: 월·목) | **요일/간격** 정보 필요 |
| 월별 | 매월 N일 또는 제N주 | **일(1~31)** 또는 **주차** 정보 필요 |
| 계절별 | 3·6·9·12월 등 | **월** 또는 **기간** 정보 필요 |

즉, **단순 주기(매일/주 1회)** 는 현재 테이블로 충분하고,  
**요일·월·계절**까지 쓰면 **추가 정보(요일, 일, 월 등)** 가 필요합니다.

---

## 3. 방안 비교

### 방안 A: 현재 테이블로 통합 (추천)

**방식**
- 시설 일정도 `schedule_items`에 `targetType: 'facility'`로 저장
- `schedule_basis_types`에 시설용 기준 추가: **매일, 주 1회, 주 2회, 월별, 계절별** 등
- “날짜”는 우선 **dayMin/dayMax를 재해석**  
  - 예: 매일=0/0, 주 1회=7/7(주기 일수), 월별=일(1~31) 등으로 코드/규칙으로 해석

**장점**
- 일정이 **한 곳**에 모여 있어 조회·캘린더·리포트가 단순함
- 대상장소·작업유형·작업내용 등 **기존 개념 그대로 재사용**
- UI(일정 관리 설정)를 **한 화면**으로 유지 가능
- 나중에 “시설만 보기” 필터만 추가하면 됨

**단점**
- 요일/월/계절 등 **복잡한 반복**은 dayMin/dayMax만으로는 한계 → 확장 필요

**적합한 경우**
- 주로 **매일, 주 1회, 주 2회, 월 1회** 수준이면 충분한 경우
- “매월 15일”, “3·6·9·12월” 정도만 필요하고, 상세 반복 규칙이 크게 늘어나지 않을 경우

---

### 방안 B: 시설만 별도 테이블/운영

**방식**
- 시설 관리용 **별도 테이블** (예: `facility_schedule_rules`)  
  - 반복 유형(일/주/월/계절), 요일, 일(1~31), 월 등 **전용 컬럼** 구성
- 일정 목록/캘린더는 **돼지 일정 + 시설 규칙**을 각각 조회 후 화면에서 합침

**장점**
- 시설 전용 **반복 규칙**을 스키마에 명확히 표현 가능
- 주 2회(요일 지정), 월별(일/주차), 계절별(월) 등 **복잡한 조건**을 무리 없이 저장

**단점**
- **일정이 두 시스템**으로 나뉨 → 조회·통계·UI 이중화
- 작업유형·대상장소 등이 **중복 정의**되거나 연동 로직 필요

**적합한 경우**
- 시설 관리가 **요일/월/계절 조합**으로 많이 쓰이고,  
  “주 2회 월·목”, “매월 둘째 주 월요일”, “분기별 1일” 등 **세밀한 규칙**이 계속 늘어날 경우

---

## 4. 추천 요약

| 상황 | 추천 |
|------|------|
| **매일, 주 1회, 월 1회 수준**이 대부분이고, “계절별”도 “3·6·9·12월” 정도만 쓰는 경우 | **방안 A: 현재 테이블 통합** |
| **요일 지정(주 2회 월·목), 매월 N일/제N주, 계절별 상세**가 핵심 요구이고 앞으로도 늘어날 경우 | **방안 B: 시설 별도 테이블** 검토 |

**실무적으로는 방안 A를 먼저 추천**합니다.

- 지금은 **기준 유형만 확장**(매일, 주 1회, 주 2회, 월별, 계절별)하고  
  dayMin/dayMax를 **시설용으로만** 다른 의미로 쓰면  
  (예: 월별일 때 dayMin = 해당 일(1~31), 계절별일 때는 코드값 등)  
  대부분의 “주 몇 회·월별·계절별” 요구를 커버할 수 있습니다.
- 나중에 **요일/월/계절**을 꼭 구분해 저장해야 할 필요가 생기면,  
  **같은 테이블에 컬럼만 추가** (예: `recurrenceWeekdays`, `recurrenceMonthDay`, `recurrenceMonths` 또는 JSON 한 컬럼) 하는 **하이브리드**로 가는 것도 가능합니다.  
  이렇게 하면 “한 테이블·한 목록”은 유지하면서 시설만 조금 더 풍부하게 다룰 수 있습니다.

---

## 5. 방안 A로 갈 때 바로 할 작업 예시

1. **기준 유형**  
   - 시설용 기준 추가: `매일`, `주 1회`, `주 2회`, `월별`, `계절별` (targetType: facility)
2. **일정 항목**  
   - 구분=시설인 일정 추가 시, 위 기준 선택 + 작업유형·대상장소·작업내용 입력
3. **표시 규칙**  
   - 시설 일정에서 dayMin/dayMax 표시 시, 기준에 따라  
     “매일”, “주 1회”, “월 N일”, “계절별(3·6·9·12월)” 등으로 **라벨만 다르게** 보여주기
4. **(선택)**  
   - 나중에 요일/월 등을 저장해야 하면 `schedule_items`에  
     `recurrenceOption`(JSON 또는 별도 컬럼들) 추가

이렇게 하면 **시설 관리 기준도 현재 테이블을 이용해 관리**하는 방향이 맞고,  
복잡도가 커질 때만 별도 테이블 또는 확장 필드를 도입하는 쪽을 추천합니다.
