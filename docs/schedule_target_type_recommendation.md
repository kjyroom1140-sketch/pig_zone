# 일정 관리 "구분" 확장 추천

현재 **구분**은 **돼지(pig)** / **시설(facility)** 두 가지입니다.  
구분을 더 나누고 싶을 때 선택할 수 있는 방향을 정리했습니다.

---

## 1. 현재 구조 요약

| 구분 | targetType | 용도 | 사용하는 필드 |
|------|------------|------|----------------|
| 돼지 | pig | 개체·돈군 기준 일정 (전입일, 출산일 등) | 기준, 일령, dayMin~dayMax |
| 시설 | facility | 시설·장소 기준 반복 일정 | 반복(매일/매주/매월), 대상장소 |

---

## 2. 추천 방향 (3가지)

### 방안 A: 구분 값만 추가 (targetType 확장)

**내용**  
`targetType`에 값을 더 추가합니다. 예: `pig`, `facility`, `feed`, `environment`, `labor` 등.

| 값(예시) | 표시명 | 용도 |
|----------|--------|------|
| pig | 돼지 | 기존과 동일 (기준일+일수, 일령) |
| facility | 시설 | 기존과 동일 (반복, 시설 유지보수) |
| feed | 사료·급여 | 사료 전환, 급여량 변경 등 (돼지와 유사하게 기준일+일수 사용 가능) |
| environment | 환경·시설관리 | 환기, 온도, 습도, 소독 등 (시설과 유사, 반복 위주) |
| labor | 인력·작업 | 담당자 배정, 작업 순서 등 (선택 사항) |

**장점**  
- 구조가 단순하고, 필터/통계 시 “구분” 하나로 묶기 좋음.  
- 일정 항목·API는 `targetType` 조건만 확장하면 됨.

**단점**  
- 값이 늘어날수록 “돼지/시설”처럼 **폼 동작**(기준일 vs 반복)을 구분할 규칙을 새로 정해야 함.  
- `feed`는 돼지와 비슷(기준+일수), `environment`는 시설과 비슷(반복)으로 보면, **동작 그룹**을 정해 두는 게 좋음.

**구현 포인트**  
- DB: `schedule_items.targetType`, `farm_schedule_items.targetType`에 새 코드 허용 (VARCHAR 길이만 넉넉히).  
- 기준 유형: `schedule_basis_types.targetType`에 `feed` 등 추가하면 “사료” 전용 기준도 구분 가능.  
- 화면: 구분 셀렉트에 옵션 추가, 선택한 구분에 따라 “기준·일령·일수” vs “반복” 노출 규칙 정의.

---

### 방안 B: 구분은 유지 + “일정 카테고리” 추가 (추천)

**내용**  
**구분(돼지/시설)**은 그대로 두고, **일정 카테고리**를 별도 필드로 둡니다.  
“대상이 돼지인지 시설인지”는 기존처럼 두고, “어떤 업무 영역인지”만 카테고리로 구분하는 방식입니다.

| 카테고리(예시) | 설명 |
|----------------|------|
| 사육관리 | 포유·이유·비육·분만·임신 등 일상 사육 일정 |
| 건강·백신 | 백신, 검역, 투약, 건강 점검 |
| 사료·급여 | 사료 전환, 급여량·급여 방식 변경 |
| 시설·환경 | 소독, 환기, 온도, 시설 점검, 정리 |
| 이동·재편 | 전동, 합사, 분리 등 돈군 이동 |
| 인력·기타 | 담당 배정, 교육, 기타 |

**장점**  
- **구분(돼지/시설)** 은 그대로라서, 현재 “기준일+일수” vs “반복” 로직을 크게 바꿀 필요 없음.  
- 필터·통계는 “구분 + 카테고리” 조합으로 세분화 가능.  
- 작업 유형(`schedule_task_types`)의 `category`를 그대로 활용할 수 있어, **새 테이블 없이** 확장 가능.

**구현 포인트**  
- **옵션 1**: `schedule_task_types.category`를 “일정 카테고리”로 쓰고, 일정 항목은 **작업유형 선택 → 카테고리 자동 반영**.  
  - 카테고리만 마스터로 정리(사육관리, 건강·백신, …)하고, 작업 유형에 카테고리 연결.  
- **옵션 2**: `schedule_items`(및 farm 쪽)에 `category` 컬럼 추가하고, “구분” 선택 다음에 “카테고리” 셀렉트를 두는 방식.  
- 화면: 목록/필터에 “카테고리” 드롭다운 추가.

---

### 방안 C: 돼지/시설 하위 유형만 추가

**내용**  
구분은 **돼지 / 시설**만 유지하고, 각각 **하위 유형**을 둡니다.

- **돼지**  
  - 생산단계: 포유, 이유, 비육, 분만, 임신, 검역 등 (선택)  
  - 또는 “기준”으로 이미 구분되어 있으므로, 하위 유형은 “표시·필터용” 정도만.
- **시설**  
  - 유형: 시설 유지보수, 환경 관리(환기/온도), 소독, 점검 등 (선택)

**장점**  
- targetType은 2개로 유지해서 기존 로직을 거의 그대로 쓸 수 있음.  
- 필터/보고에서 “돼지 중 포유만”, “시설 중 소독만”처럼 더 잘게 나눌 수 있음.

**단점**  
- 하위 유형용 테이블·셀렉트가 하나 더 필요함.  
- “사료”처럼 돼지이면서 시설(사료 공급 설비)에 가까운 경우는 구분이 모호해질 수 있음.

**구현 포인트**  
- `schedule_items`에 `targetSubType` 또는 `scheduleCategory`(VARCHAR) 추가.  
- 마스터는 코드 테이블 하나(schedule_target_sub_types) 또는 상수 목록으로 관리.

## 3. 정리 및 추천

| 방안 | 변경 규모 | 추천 상황 |
|------|-----------|-----------|
| **A** targetType 확장 | 중간 (값 추가 + 폼 규칙) | “사료”, “환경”, “인력”을 **구분** 자체로 다루고 싶을 때 |
| **B** 구분 유지 + 카테고리 | 작음 (필터/표시 위주) | “돼지/시설”은 그대로 두고, **업무 영역(사육/건강/사료/시설 등)** 만 나누고 싶을 때 |
| **C** 하위 유형 | 작음 (컬럼 1개 + 마스터) | “돼지/시설” 안에서만 **세부 유형**으로 필터·표시하고 싶을 때 |

**실무 추천**  
- **우선은 방안 B**를 추천합니다.  
  - 구분(돼지/시설)과 현재 “기준일 vs 반복” 동작을 유지한 채,  
  - **작업 유형의 category** 또는 **일정 항목용 category** 하나로 “사육/건강/사료/시설/이동” 등을 나누면,  
  - DB·API 변경을 줄이면서도 목록/필터를 세분화할 수 있습니다.  
- 나중에 “사료만 따로 기준일·일수를 다르게 쓰고 싶다” 등 **동작이 구분마다 달라져야 하면** 방안 A를 검토하는 식으로 확장하는 것이 좋습니다.

---

## 4. 적용 시 공통 확인 사항

- **기준 유형** (`schedule_basis_types`): 새 구분/카테고리를 쓸 때 `targetType`(또는 새로 두는 카테고리)와 맞는 기준만 노출할지 정리.  
- **작업 유형** (`schedule_task_types`): 이미 `category`가 있으므로, “일정 카테고리”와 매핑만 정하면 됨.  
- **필터/그리드**: admin·farm_admin·대시보드 일정 화면에 “구분” 다음에 “카테고리” 또는 “하위 유형” 필터 추가.

원하시는 방향( A / B / C 또는 조합)이 정해지면, 그에 맞춰 `targetType`/컬럼/화면 변경 포인트를 더 구체적으로 적어 드리겠습니다.

## 5. 돼지 구분: 모돈 / 옹돈 / 나머지 (대상 돼지 유형)

**요구**  
- **시설**은 그대로 두고, **돼지** 일정만 “대상이 어떤 돼지인지”로 나누고 싶다.  
- 예: **모돈** / **옹돈** / **그 나머지(일반돼지 등) 통틀어** 한 단계로 구분.

### 5.1 구분 구조 제안

| 구분(기존) | 돼지 대상(신규) | 설명 |
|------------|----------------|------|
| 시설 | (해당 없음) | 기존과 동일 |
| 돼지 | **모돈** | 번식용 암퇘지 (분만·포유·임신·교배 등) |
| 돼지 | **옹돈** | 번식용 수퇘지 (종돈) |
| 돼지 | **(나머지)** | 모돈·옹돈 제외한 모든 돼지 (아래 명칭 참고) |

- **시설**은 변경 없이 그대로 두고, **돼지**일 때만 “돼지 대상”을 **모돈 / 옹돈 / 나머지** 중에서 선택하는 방식이면, 기존 로직(기준일·일수·반복)을 유지하기 좋습니다.

### 5.2 “일반돼지” 대신 쓸 만한 명칭 추천

“나머지”(모돈·옹돈 제외)를 한 칸에 넣을 때 **일반돼지**보다 업계에서 많이 쓰는 말을 정리하면 아래와 같습니다.

| 명칭 | 설명 | 비고 |
|------|------|------|
| **비번식돈** | 번식에 쓰지 않는 돼지 (모돈·옹돈 제외) | 의미가 가장 명확함. **추천** |
| **육성돈** | 자돈~비육 등 성장·비육 단계 통틀어 | “육성” 강조할 때 적합 |
| **자돈·비육돈** | 포유·이유·비육 단계를 문구로 표현 | 구체적이지만 항목명이 김 |
| **사육돈** | 사육 중인 돼지 전반 (넓은 의미) | 모돈·옹돈도 포함될 수 있어 “나머지만” 지칭하기엔 다소 모호 |
| **번식외 돼지** | 번식용이 아닌 돼지 | 비번식돈과 유사, 조금 격식 있는 표현 |

**정리**  
- **비번식돈**: “모돈·옹돈이 아닌 나머지”를 한 단어로 쓰기 좋고, **가장 추천**합니다.  
- “육성·비육” 느낌을 살리고 싶으면 **육성돈**을 두 번째 후보로 쓸 수 있습니다.

### 5.3 구현 방향 (요지)

- **구분**은 그대로 **돼지 / 시설** 두 가지.  
- **돼지**를 선택했을 때만 **“돼지 대상”** 선택 추가: **모돈 / 옹돈 / 비번식돈**(또는 육성돈 등).
- DB: `schedule_items`, `farm_schedule_items`에  
  `pigTargetType`(또는 `targetPigCategory`) VARCHAR  
  예: `sow`(모돈), `boar`(옹돈), `non_breeding`(비번식돈).  
  NULL 또는 빈 값이면 “전체/미지정”으로 두어도 됨.
- 목록·필터: 구분(돼지/시설) 다음에 “돼지 대상”(모돈/옹돈/비번식돈) 필터 추가.
- 기준·작업유형: 필요하면 “돼지 대상”별로 노출할 항목만 제한하는 규칙을 나중에 추가 가능.
