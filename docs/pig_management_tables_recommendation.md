# 돼지 관리 테이블 구성 방향 — 객체 / 돈군 / 이동

돼지 관리 시 **객체(개체) 테이블**, **돈군 테이블**, **이동 관리 테이블**을 어떻게 둘지에 대한 권장 방향입니다.

---

## 1. 세 테이블의 역할 정리

| 테이블 | 역할 | 비고 |
|--------|------|------|
| **객체(개체) 테이블** | 한 마리 단위 관리 (이력, 혈통, 개체 ID 등) | 필요 여부는 농장 운영 방식에 따라 다름 |
| **돈군 테이블** | “한 무리” 단위 관리 (두수, 입식일, 일령, 품종, 현재 칸 등) | 일정·이동·사육 관리의 **기본 단위**로 권장 |
| **이동 관리 테이블** | “언제, 어디서 → 어디로, 몇 두(또는 어느 돈군)” 기록 | 이전 권장안대로 **별도 테이블** 권장 |

---

## 2. 권장 방향 요약

- **돈군 테이블**: 두고, **일정·이동·사육 관리의 기본 단위**로 사용하는 것을 권장.
- **이동 관리 테이블**: 두고, **이동 이벤트**만 전담하도록 두는 것을 권장.
- **객체(개체) 테이블**: **선택**. 개체별 추적(어미돼지, 종돈, 개체 ID·혈통 등)이 필요하면 추가.

즉, **최소 구성은 “돈군 + 이동”**, 필요하면 “객체 + 돈군 + 이동”으로 확장하는 방향을 추천합니다.

---

## 3. 왜 돈군 테이블이 필요한가

- 현재 **farm_sections(칸)** 에 `currentPigCount`, `entryDate`, `birthDate` 등이 있는데, 이는 **“이 칸에 지금 있는 무리의 요약”**에 가깝습니다.
- 문제점:
  - **이동 시** “어느 무리가 옮겨 갔는지”를 구분하기 어렵고,
  - 같은 칸에 **시점이 다른 여러 무리**가 들어온 경우(이력)를 구분하기 어렵습니다.
- **돈군을 독립 엔티티**로 두면:
  - 한 돈군 = 한 레코드 (두수, 입식일, 품종, 현재 위치(section_id), 출하일 등)
  - 이동 시 “돈군 A가 칸 1 → 칸 2로 이동”으로 기록 가능
  - 일정(스케줄)도 “어느 돈군/어느 시설 대상”으로 연결하기 쉬움

→ **칸(시설)** = “장소”, **돈군** = “그 장소에 있는 사육 단위(무리)” 로 분리하는 구성을 권장합니다.

---

## 4. 객체(개체) 테이블은 언제 두는가

- **돈군 단위만** 관리하면: 개체 테이블 없이도 “이 칸/이 돈군 몇 두, 입식일, 일령” 수준의 운영이 가능합니다.
- **개체 단위** 관리가 필요한 경우 예:
  - 어미돼지·종돈 등 **개체별 이력·혈통·산차** 관리
  - 개체별 **백신·질병·이동** 이력 추적
  - 개체 ID(귀표 등)로 한 마리 식별

→ 이런 요구가 있으면 **객체(개체) 테이블**을 추가하고,  
- 개체가 “어느 돈군에 속하는지”(돈군_id 또는 현재 section_id)  
- 이동 시 “개체 단위 이동”을 이동 테이블에 기록할지, “돈군 단위만” 기록할지  

를 정책으로 나누면 됩니다.  
**초기에는 돈군+이동만 두고, 개체 관리 요구가 생기면 객체 테이블을 추가**하는 단계적 구성을 추천합니다.

---

## 5. 테이블 간 관계 권장

```
[객체 테이블]  (선택)
     │  현재 소속
     ▼
[돈군 테이블]  ◄──── 현재 위치: farm_sections.id (current_section_id)
     │
     │  이동 이벤트
     ▼
[이동 관리 테이블]  ──  from_section_id, to_section_id, pig_group_id(돈군), moved_at, headcount, schedule_item_id 등
     │
     └── 시설(farm_sections)과 연결
```

- **돈군**  
  - `current_section_id` (현재 있는 칸)  
  - 두수, 입식일, 일령, 품종, 출하일 등  
- **이동**  
  - “돈군” 또는 “두수+출발/도착 칸” 단위로 1건 1행  
  - 선택: `schedule_item_id`로 일정과 연결  
- **객체**  
  - 있는 경우: 현재 소속 돈군 또는 현재 section, 개체별 이력

---

## 6. 단계별 도입 제안

| 단계 | 내용 |
|------|------|
| **1단계** | **돈군 테이블** 도입 — 칸별 “현재 두수·입식일” 등을 돈군 레코드로 이전·정리. 칸은 “위치”만, 돈군은 “사육 단위”만 담당. |
| **2단계** | **이동 관리 테이블** 도입 — 이동 시 돈군 단위(또는 두수+출발/도착 칸)로 1건 1행 기록, 돈군의 current_section_id 갱신. |
| **3단계** | 개체 관리 요구가 있으면 **객체 테이블** 추가 — 어미/종돈·개체 이력 등. 이동은 “돈군 단위” 위주로 두고, 필요 시 개체 이동만 추가 컬럼/테이블로 확장. |

---

## 7. 정리

- **돈군 테이블**: 두는 것을 권장. 일정·이동·사육의 **기본 관리 단위**로 사용.
- **이동 관리 테이블**: 두는 것을 권장. **이동 이벤트 전용**으로 두고, 돈군(또는 두수)과 출발/도착 시설을 연결.
- **객체 테이블**: **선택**. 개체별 추적이 필요할 때 추가하고, 없으면 “돈군 + 이동”만으로도 운영 가능.

이렇게 하면 “객체 – 돈군 – 이동” 역할이 나뉘어 확장과 조회·감사 모두 유리합니다.

---

## 8. 객체 테이블에 돈군 컬럼을 두어 관리하는 방법

**객체(개체) 테이블**에 **돈군을 가리키는 컬럼 하나**를 두면, 개체 단위와 돈군 단위를 같이 다룰 수 있습니다.

### 8.1 구조

- **객체 테이블** (예: `pigs`, `animal_individuals`)
  - 기존 개체용 컬럼 (id, 개체번호, 품종, 성별, 출생일, …)
  - **추가 컬럼**: `pig_group_id` (NULL 가능, FK → 돈군 테이블)
- **돈군 테이블** (예: `pig_groups`)
  - id, farm_id, current_section_id, 입식일, 품종(대표), 비고, …
  - 두수는 **해당 돈군에 속한 객체 수**로 계산 가능 (COUNT)

관계: **돈군 1 : 객체 N** (한 돈군에 여러 개체가 소속).

```
[돈군 pig_groups]  1  ──────<  N  [객체 pigs]
       │                              │
       │ id                           │ pig_group_id (FK)
       ▼                              ▼
 current_section_id              개체별 정보
 입식일, 비고 등                  + 소속 돈군
```

### 8.2 컬럼 추가 예시 (객체 테이블)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| pig_group_id | INTEGER NULL FK → pig_groups.id | 이 개체가 속한 돈군. NULL이면 미편입 또는 돈군 미사용 |

- 개체를 돈군에 넣을 때: 해당 개체의 `pig_group_id`를 그 돈군의 id로 설정.
- 돈군에서 빼거나, 돈군 없이 관리할 때: `pig_group_id = NULL` 또는 다른 돈군 id로 변경.

### 8.3 이렇게 두었을 때 장점

- **돈군 단위 조회**: "이 돈군에 속한 개체 목록" → `WHERE pig_group_id = ?`
- **개체 단위 조회**: "이 개체가 어느 돈군/어느 칸에 있는지" → 객체의 `pig_group_id` → 돈군의 `current_section_id`
- **이동**: 돈군 단위로 이동 시, 해당 돈군의 `current_section_id`만 바꾸면 되고, 개체 테이블은 `pig_group_id`만 유지해도 됨.
- **두수**: 돈군별 두수 = `COUNT(*) ... WHERE pig_group_id = ?` 또는 돈군 테이블에 headcount를 두고 이동/입식 시 갱신.

### 8.4 정리

- **객체 테이블에 `pig_group_id` 하나 추가**하면 돈군 테이블과 연결해 "객체–돈군"을 함께 관리할 수 있습니다.
- 돈군 테이블은 그대로 두고, 객체가 "어느 돈군 소속인지"만 FK로 가지는 구조이므로, 앞에서 권장한 "돈군 + 이동" 설계와도 잘 맞습니다.

## 9. 한 칸에 여러 돈군이 섞이는 경우 + 돈군 ID 부여 방법

자돈에서 돈군을 기록하다가 **한 돈군이 둘로 나뉘어 다른 방으로 이동**하면, 도착 칸에는 **원래 있던 돈군 + 새로 들어온 돈군**이 함께 있게 됩니다.  
"이 칸에 어느 돈군과 어느 돈군이 있다"를 기록하는 것은 의미 있지만, **돈군이 분할·혼합될 때 ID를 어떻게 부여할지**가 혼란스러울 수 있어, 아래 원칙을 추천합니다.

### 9.1 한 칸에 여러 돈군이 있는 경우

- **한 칸(방)에 동시에 여러 돈군**이 있을 수 있음 (예: A돈군 5두 + B돈군 3두).
- 따라서 "돈군 1개 = 현재 위치 1개(칸 1개)"가 아니라, **"칸 1개 = 돈군 여러 개(각각 두수)"** 관계를 허용하는 편이 좋습니다.

**권장: 칸–돈군 배치 테이블 도입**

| 테이블 (예시명) | 역할 |
|----------------|------|
| **section_group_occupancy** (칸–돈군 배치) | "어느 칸에, 어느 돈군이, 몇 두 있는지" 현재 시점 기록. 한 칸에 여러 행(돈군별 1행) 가능. |

예시 컬럼: `id`, `section_id`, `pig_group_id`, `headcount`, `started_at`, `ended_at`(해당 칸에서 나갔을 때) 등.  
"지금 이 칸에 어떤 돈군들이 있는지"는 `section_id`로 조회하면 됩니다.  
돈군 테이블의 `current_section_id`는 "주된 위치" 하나만 둘지, 아니면 배치 테이블만 두고 돈군은 "현재 어디 있는지"를 배치 테이블로만 파악할지 정책으로 나누면 됩니다.

### 9.2 돈군이 둘로 나뉠 때(분할 시) ID 부여 원칙

- **분할 = 기존 돈군을 "종료"하고, 나뉜 결과를 "새 돈군"으로 생성**하는 방식이 단순하고 혼란을 줄입니다.

| 단계 | 처리 |
|------|------|
| 1 | 원 돈군(예: ID=100)을 **분할됨/종료** 상태로 변경 (status='split' 또는 'closed', 종료일 기록). |
| 2 | **새 돈군 2개**를 생성 → 각각 **새 ID** 부여 (예: 101, 102). 생성 시마다 자동 증가 ID 또는 UUID 사용. |
| 3 | 객체들: 원래 돈군 100에 속하던 개체를 **새 돈군 101 또는 102**에 배정 → 각 객체의 `pig_group_id`를 101 또는 102로 변경. |
| 4 | 이동·배치: 새 돈군 101은 칸 A로, 102는 칸 B로 이동했다면, 이동 테이블·배치 테이블에 그대로 기록. |

**정리: 돈군 ID는 "한 번 생성된 무리의 identity"로만 씁니다.**  
- 새로 만들 때마다 **새 ID 1개** 부여.  
- **나뉘면 원 돈군은 더 이상 "현재 존재하는 그룹"이 아니므로** 종료/분할 표시만 하고, **나뉜 각 무리는 새 돈군(새 ID)** 로만 관리합니다.  
- "이 돈군이 저 돈군에서 나뉜 것"을 남기고 싶으면, 돈군 테이블에 **parent_group_id** (NULL 가능) 컬럼을 두어, 101·102가 "원래 100에서 분할"임을 참조할 수 있습니다.

### 9.3 ID 부여 방법 요약 (혼란 방지)

| 상황 | ID 부여 / 처리 |
|------|----------------|
| **새 돈군 생성** (입식, 전입 등) | 새 레코드 1개 생성 → **새 ID 1개** (자동 증가 또는 UUID). |
| **돈군 분할** (한 돈군이 둘로 나뉨) | 원 돈군 → 종료/분할 처리. **나뉜 각 무리마다 새 돈군 1개 생성 → 새 ID 각 1개.** (총 2개 나뉘면 ID 2개.) |
| **돈군 합병** (두 돈군을 한 칸에서 한 무리로 뭉침) | 한쪽 돈군을 "종료"하고, 다른 한 돈군에 객체들을 모두 소속시키거나, "합병된 새 돈군" 1개를 만들어 새 ID 1개 부여 후 소속 정리. 정책에 따라 선택. |
| **한 칸에 여러 돈군** | 돈군 ID는 그대로 두고, **"이 칸에 돈군 A(두수 n), 돈군 B(두수 m)"** 는 **칸–돈군 배치 테이블**에 기록. |

- **객체의 pig_group_id**: 어떤 시점이든 **반드시 하나의 돈군 ID만** 가리키도록 합니다. 분할 시에는 위처럼 새 돈군 ID 중 하나로 갱신하면 됩니다.
- **이력·추적**: "원래 100에서 101·102로 나뉜 것"은 `parent_group_id`로, "어느 칸에 언제 있었는지"는 이동 테이블·배치 테이블로 조회하면 됩니다.

이렇게 하면 **"돈군 ID = 한 번 부여된 identity, 분할 시에는 새 ID만 추가"** 로 통일되어, 객체가 섞이고 나뉘어도 ID 부여가 혼란스럽지 않습니다.
