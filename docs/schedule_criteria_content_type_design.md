# 기준 선택에 따른 기준내용 입력 조건 설계

## 목표
- **현재**: 기준내용에서 "유형"을 직접 선택 → 유형별로 시작일/종료일, 매일, 횟수 등 입력
- **변경**: **기준**에서 선택한 항목(출생일·횟수·매일·주말 등)에 따라 기준내용 입력 형태가 자동 결정

## 제안: 기준(criteria)에 "기준내용 유형" 저장

### 1. 데이터 구조

**schedule_criterias** 테이블의 `criterias` JSONB에 **content_type**을 포함합니다.

- 현재: `criterias` = `[{ "name": "출생일(일령)" }]` 형태
- 변경: `criterias` = `[{ "name": "출생일(일령)", "content_type": "range" }]` 형태

**content_type 값 (기준내용 입력 형태)**

| content_type | 기준 예시 이름   | 기준내용에 노출되는 입력                    |
|--------------|------------------|---------------------------------------------|
| `range`      | 출생일(일령)     | 시작일, 종료일 (일 단위)                    |
| `count`      | 횟수             | 입력창 + "회"                               |
| `daily`      | 매일             | "매일" 표시만 (추가 입력 없음)              |
| `weekend`    | 주말             | 주말용 시작일·종료일 또는 "주말"만          |
| `monthly`    | 월단위           | 매월 몇 일                                  |
| `yearly`     | 년 1회           | 매년 월·일                                  |
| `weekly`     | N주마다 요일     | 몇 주마다 + 요일 체크                       |

- 기존 데이터에 `content_type`이 없으면 **기본값 `range`** 로 처리하면 됩니다.

### 2. UI/플로우 변경

1. **기준 추가/수정 시**
   - "기준 이름" 입력 + **"기준내용 유형"** 선택(드롭다운) 추가  
     - 예: 시작일·종료일(출생일/일령), 횟수, 매일, 주말, 월단위, 년 1회, N주마다 요일
   - 저장 시 `criterias` JSON에 `content_type` 포함  
     - 예: `[{ "name": "출생일(일령)", "content_type": "range" }]`

2. **기준내용 영역**
   - **유형 셀렉트 제거**: 사용자가 유형을 고르지 않음
   - **선택된 기준의 content_type**만 보고:
     - `range` → 시작일·종료일 입력만 표시
     - `count` → 횟수 입력 + "회" 표시
     - `daily` → "매일" 문구만
     - `weekend` → 주말용 시작일·종료일 또는 "주말"만
     - `monthly` → 매월 몇 일
     - `yearly` → 매년 월·일
     - `weekly` → N주마다 + 요일

3. **저장**
   - `schedule_work_plans.criteria_content`는 **지금처럼**  
     `{ type: 'range', start_day, end_day }`, `{ type: 'count', count }` 등으로 저장
   - **type**은 선택된 기준의 **content_type**과 동일하게 넣으면 됨 (기준 1개 = 유형 1개)

### 3. 구현 단계 제안

| 단계 | 내용 |
|------|------|
| 1 | **기준 API/목록**: `criterias`에서 첫 항목의 `content_type` 읽어서 목록/상세에 포함. 없으면 `'range'` 사용 |
| 2 | **기준 추가/수정 폼**: "기준 이름" + "기준내용 유형" 선택 추가, 저장 시 `content_type` 포함 |
| 3 | **기준내용 영역**: 유형 셀렉트 제거. `criteriaRows[selectedCriteriaIndex].contentType`으로 표시할 블록 결정 (시작일·종료일 / 횟수+회 / 매일 / 주말 / 월단위 / 년 1회 / N주마다 요일) |
| 4 | **buildCriteriaContent()**: 선택된 기준의 `contentType`을 `type`으로 사용하고, 해당 입력값만 담아서 저장 |

### 4. 대안: 이름으로 유형 추론 (DB 변경 없음)

- `content_type` 컬럼/필드 없이, **기준 이름(라벨)**으로 유형을 추정
  - "출생일", "일령" 포함 → `range`
  - "횟수" 포함 → `count`
  - "매일" → `daily`
  - "주말" → `weekend`
  - "월단위", "월" 포함 → `monthly`
  - "년", "1회" 등 → `yearly`
  - "주마다", "요일" 등 → `weekly`
- **장점**: DB/API 변경 없이 프론트만 수정 가능  
- **단점**: 이름 규칙에 강하게 묶이고, 다국어·이름 변경 시 깨지기 쉬움  

이미 기준 이름을 자유롭게 쓰고 있다면 **제안안(기준에 content_type 저장)** 이 더 안전하고 확장하기 좋습니다.

### 5. 요약

- **권장**: 기준(criterias JSON)에 `content_type`을 두고, **기준 선택 → 해당 content_type에 맞는 기준내용 입력만 노출**하도록 변경
- 기준내용 유형 셀렉트는 제거하고, **기준 추가/수정 시**에만 "기준내용 유형"을 선택하게 하면, "출생일(일령) → 시작일·종료일", "횟수 → 횟수 입력 + 회", "매일 → 매일" 등이 자연스럽게 연결됩니다.

이 설계대로 적용할지, 아니면 먼저 "이름으로 유형 추론" 방식으로 빠르게 붙일지 정하시면 그에 맞춰 다음 수정 단계(어디부터 손댈지)까지 구체화해 드리겠습니다.
