# 일정 마스터 — 작업유형이 "이동"일 때 대상 사육시설 선택 추천

> 작업유형이 **이동**인 경우, **어느 사육시설로 이동**인지 일정 마스터에서 선택하는 UI를 추가할 때, DB·API·UI를 어떻게 처리할지 추천합니다.

---

## 1. 현재 구조

- **schedule_work_plans** 에는 **structure_template_id** 가 있습니다.  
  → 이 작업 계획이 **어느 시설에 적용되는지**(출발/작업 대상 시설)를 나타냅니다.
- **이동**인 경우, "그 시설에서 **어느 시설로** 이동하는지"는 **저장하는 곳이 없습니다**.

따라서 "이동 대상 사육시설"을 **어디에 어떻게 저장할지**만 정하면 됩니다.

---

## 2. 추천: 컬럼 1개 추가 (target_structure_template_id)

**권장:** `schedule_work_plans` 에 **컬럼 1개**만 추가하는 방식입니다.

| 컬럼명 | 타입 | NULL | 설명 |
|--------|------|------|------|
| **target_structure_template_id** | INTEGER | Y | 이동 대상 시설 → structure_templates.id (FK). 작업유형이 "이동"일 때만 사용, 그 외는 NULL. |

**장점**

- 의미가 명확하고, 조회·조인·인덱스가 단순함.
- "이동"일 때만 채우고 나머지는 NULL 이므로, 기존 행·다른 작업유형에 영향 없음.
- UI는 "작업유형 = 이동"일 때만 "이동 대상 시설" 드롭다운을 노출하면 됨.
- 작업 목록 테이블(하단 그리드)은 **컬럼을 많이 늘릴 필요 없음**.  
  - 옵션 A: "이동 대상" 컬럼 1개만 추가해서, 이동인 행만 값 표시.  
  - 옵션 B: 목록에는 안 넣고, 상단 폼에서만 "이동 대상 시설" 선택 후 저장.

**마이그레이션 예시**

```sql
ALTER TABLE schedule_work_plans
  ADD COLUMN IF NOT EXISTS target_structure_template_id INTEGER REFERENCES structure_templates(id);
```

**API**

- Create/Update 시 body 에 `target_structure_template_id` (optional) 추가.
- List/Get 시 응답에 `target_structure_template_id`, 필요 시 `target_structure_template_name` (JOIN) 포함.

**UI (일정 마스터)**

- 상단: 시설 → 구분 → **작업유형** 선택 후, **작업유형이 "이동"일 때만** "이동 대상 시설" 드롭다운 표시.
- 저장 시 이동이면 `target_structure_template_id` 전송, 그 외에는 null/미전송.
- 하단 작업 목록: "이동"인 행에만 "이동 대상" 칸에 시설명 표시(선택).

---

## 3. 대안: JSONB 로 처리 (컬럼 수 유지)

테이블 컬럼을 **늘리지 않고**, 작업유형별 옵션을 JSONB 로만 넣고 싶다면 다음처럼 할 수 있습니다.

**방법 A — 기존 컬럼 활용**

- **criteria_content** 에 이동일 때만 `move_target_template_id` 를 넣는 방식.  
  - 장점: 스키마 변경 없음.  
  - 단점: 기준내용과 이동 대상이 한 컬럼에 섞여서, 의미가 불명확해지고 조회/검증이 번거로움. **비권장.**

**방법 B — 전용 JSONB 컬럼 1개 추가**

- `schedule_work_plans` 에 **jobtype_options** (JSONB, NULL) 추가.  
  - 이동일 때: `{ "move_target_template_id": 2 }`  
  - 그 외: NULL 또는 `{}`
- 장점: 컬럼은 1개만 추가하고, 나중에 다른 작업유형용 옵션(예: 분만 시 옵션)도 같은 컬럼에 확장 가능.
- 단점: FK·제약으로 "유효한 시설 id 인지"를 DB 레벨에서 보장하려면 CHECK/트리거가 필요하고, 조인 시 JSONB 언박싱이 들어감.

**정리:**  
- **스키마를 한두 개만 추가하고, 의미를 분명히 하고 싶으면** → **2절의 `target_structure_template_id` 컬럼 1개 추가**를 추천.  
- **컬럼은 반드시 0개로 두고, 확장성만 JSONB 로 가져가고 싶으면** → 방법 B(jobtype_options)를 고려할 수 있으나, 운영·조회 단순성은 2절이 더 낫습니다.

---

## 4. 별도 테이블은 불필요

- "이동 대상"이 **작업 계획당 1개**이면 1:1 관계이므로, `schedule_work_plans.target_structure_template_id` 한 컬럼이면 충분합니다.
- 별도 테이블(예: schedule_work_plan_move_targets)은 1:N(한 작업에 대상 시설 여러 개)이 필요할 때나 고려하면 됩니다.  
  지금 요구사항("어느 사육시설로 이동인지 **하나** 선택")에는 **컬럼 1개 추가**로 충분합니다.

---

## 5. 요약

| 방식 | 테이블 컬럼 | 추천도 | 비고 |
|------|-------------|--------|------|
| **target_structure_template_id 1개 추가** | 1개 증가 | ⭕ 권장 | 의미 명확, 조회·API·UI 단순. 작업 목록은 칸 1개만 선택 추가하거나, 상단 폼에서만 입력. |
| jobtype_options (JSONB) 1개 추가 | 1개 증가 | △ 가능 | 확장성 좋음, 조인/제약은 컬럼보다 번거로움. |
| criteria_content 에 포함 | 0개 | ✗ 비권장 | 기준내용과 혼재, 스키마 오염. |
| 별도 테이블 | 새 테이블 | ✗ 불필요 | 1:1이면 컬럼으로 충분. |

**결론:**  
- **테이블 컬럼을 1개 늘려서** `target_structure_template_id` 를 두고,  
- **작업유형이 "이동"일 때만** 일정 마스터 UI에 "이동 대상 사육시설" 선택란을 추가하는 방식이 가장 단순하고 유지보수하기 좋습니다.  
- "테이블 칼럼을 늘리지 않고" 하려면 JSONB(jobtype_options)로 갈 수는 있으나, 운영 단순성과 명확성은 컬럼 추가가 더 낫습니다.
