# 돼지 이동 정보 기록 — 설계 방향 권장

나중에 **돼지 이동 정보**를 기록할 때, 스케줄·돼지·시설 중 어디에 두는 것이 맞는지 정리한 권장안입니다.

---

## 1. 세 가지 후보

| 방안 | 설명 |
|------|------|
| **A. 돼지 객체에 기록** | 돼지(또는 돼지 그룹/배치) 테이블에 이동 이력 필드 보유 |
| **B. 시설 테이블에 기록** | farm_sections, farm_rooms 등 시설·칸 테이블에 이동 관련 필드 보유 |
| **C. 별도 이동관리 테이블** | 이동 전용 테이블(pig_movements 등)에서 이력 관리 |

---

## 2. 각 방안의 특징

### A. 돼지 객체에 기록

- **장점**
  - “이 돼지 지금 어디 있지?”는 해당 레코드만 보면 됨.
  - 구현이 단순해 보일 수 있음.
- **단점**
  - **이동은 한 건이 아니라 여러 번 발생** → 이력을 배열/JSON으로 넣으면 조회·집계가 어렵고, 결국 “이동 1건 = 1행” 구조가 필요해짐.
  - “이 칸/이 돈사에서 이번 주에 일어난 이동 전부” 같은 **시설·기간 기준 조회**가 돼지 테이블만으로는 비효율적.
  - 돼지 테이블이 “현재 상태 + 이력”이 섞여 비대해지고, 책임이 불명확해짐.

→ **이동 “이력”까지 돼지 테이블에 넣는 것은 비추천.**  
다만 **“현재 위치”만** 돼지(또는 배치)에 두는 것은 권장(아래 C와 함께 사용).

---

### B. 시설 테이블에 기록

- **장점**
  - “이 칸 기준으로 무슨 일이 있었지?”를 시설 쪽에서 접근할 수 있음.
- **단점**
  - **이동은 “출발 시설 ↔ 도착 시설” 관계**라서, 한 쪽 시설에만 넣으면 비대칭이고 “반대편 시설” 조회가 불편함.
  - 시설 테이블은 **구조/정적 정보**(칸, 돈사, 방 등)용이 적합하고, **이벤트(언제 누가 어디로 이동)**는 다른 개념임.
  - 이동 이력을 시설 컬럼에 넣으면 시설 테이블이 이벤트 저장소처럼 바뀌어 설계가 흐려짐.

→ **이동 이력을 시설 테이블에 직접 넣는 것은 비추천.**

---

### C. 별도 이동관리 테이블 (권장)

- **장점**
  - **이동 = 이벤트**이므로 “이동 1건 = 1행”으로 모델링하기 좋음.
  - **출발·도착·일시·사유·두수 등**을 한 곳에서 관리할 수 있음.
  - 기간/시설/돼지(또는 배치) 기준 **조회·집계·감사(audit)** 가 단순함.
  - 일정(schedule_items)과 연계할 때 **schedule_item_id** 등으로 “어떤 일정에 따라 이동했는지” 연결하기 쉬움.
  - 돼지·시설 테이블은 각각 “현재 상태”“구조”에만 집중할 수 있음.
- **단점**
  - “현재 위치”를 알려면 “가장 최근 이동”을 조회해야 함 → 이 부분은 **돼지(또는 배치) 테이블에 current_section_id 같은 필드를 두어 보완**하는 방식을 추천.

→ **실제 이동 이력은 별도 이동관리 테이블, “현재 위치”만 돼지/배치 쪽에 두는 방식**을 권장.

---

## 3. 권장 구조 요약

- **이동 이력(언제, 어디서→어디로, 몇 두, 사유 등)**  
  → **별도 테이블**로 관리  
  - 예: `pig_movements` 또는 `farm_movements`  
  - 예시 컬럼: id, pig_id 또는 batch_id, from_section_id, to_section_id, moved_at, schedule_item_id(선택), moved_by, headcount, memo, created_at 등  

- **“지금 이 돼지/배치가 어디 있는지”**  
  → **돼지(또는 배치) 테이블**에 **현재 위치만** 저장  
  - 예: `current_section_id` (또는 current_room_id 등)  
  - 이동 발생 시: 이동 테이블에 1건 insert + 돼지/배치의 current_section_id update  

- **일정(schedule_items)**  
  → **“언제·어디로 이동해야 한다”는 규칙**만 보관  
  - 실제 “오늘 10두 분만사→이유사로 이동했다”는 **실적**은 이동 테이블에 기록  
  - 선택적으로 이동 레코드에 `schedule_item_id`를 넣어 “어떤 일정에 따라 실행한 이동인지” 연결  

- **시설 테이블(farm_sections 등)**  
  → **구조/정적 정보**만 유지, 이동 이력은 저장하지 않음  

---

## 4. 차후 설계 시 참고할 점

1. **일정(스케줄) ≠ 실적**  
   - 일정: “이동” 같은 작업 유형·대상·기준일·반복 등 **규칙**.  
   - 이동 테이블: **실제로 일어난 이동 이벤트**를 남김.  

2. **돼지 단위 vs 배치(그룹) 단위**  
   - 개체 단위 관리가 필요하면 `pig_id`, 두수 단위면 `batch_id` + `headcount` 등으로 설계하면 됨.  
   - 이동 테이블은 “1회 이동 = 1행”을 기본으로 두고, pig_id/batch_id를 선택하는 방식으로 확장 가능.  

3. **현재 위치 캐시**  
   - 매번 “최근 이동”으로 현재 위치를 계산하면 부하가 커질 수 있으므로,  
   - **돼지/배치 테이블의 current_section_id(또는 유사 필드)** 를 두고, 이동 시 갱신하는 방식을 권장.  

이 방향으로 두면 스케줄 관리·돼지 정보·시설 정보가 각각 역할이 분리되고, 이후 이동 이력 조회·리포트·감사까지 설계하기 좋습니다.
