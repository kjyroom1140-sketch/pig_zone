# 농장 일정 등록 및 작업 계획·완료 관리 진행 문서

현재는 **농장 일정 스케줄(farm_schedule_items)에 의한 화면 표시**만 구현된 상태입니다.  
이 문서는 **농장에서 직접 일정을 등록·수정·삭제**하고, **작업 계획(예정 일자)과 완료 일자**를 **별도 테이블**에서 관리하기 위한 **단계**와 **방식**을 정리한 것입니다.

**설계 방침**  
- **일정 정의**: `farm_schedule_items` — “무슨 작업을, 어떤 조건(대상장소·기준·dayMin~dayMax 등)으로 할지”만 **정의**.  
- **작업 계획·완료**: **별도 테이블** — “어느 일정을, **언제부터 언제까지** 할 예정인지”(예정 시작일·종료일)와 “**언제 완료했는지**”(완료 일자)를 한 테이블에 기록.  
  - 작업 예정은 상황에 따라 며칠 단위로 잡히며, 스케줄에도 시작·끝(dayMin/dayMax 등)이 있으므로 **예정 시작일·예정 종료일**을 둔다.  
  - **작업 계획**은 **상황 이벤트 발생 시 즉시** 생성되거나, 사용자가 특정 날짜에 일정을 추가할 때 생성된다(필요 시 해당 내용으로 다음 날짜 일정도 함께 등록).  
→ 일정(정의)과 작업 계획·완료(예정 일자 + 완료 일자)를 분리해 관리하는 방식으로 문서 전체가 일관되게 진행됩니다.

---

## 1. 현재 상태 요약

| 구분 | 내용 |
|------|------|
| **표시** | 대시보드 **일정(작업) 관리 페이지**는 그리드 표시 데이터를 **`farm_schedule_work_plans`**에서 불러와 예정 구간·완료 여부를 셀에 표시. 일정 정의(작업명·대상 등)는 `farm_schedule_items`와 JOIN하여 라벨 표시. (접힘/펼침, 돼지 유무, 검색, 한 달 보기 등 적용.) |
| **데이터** | 일정 항목(`farm_schedule_items`)은 **운영 시설 저장 시** 전역 템플릿에서 농장별로 복사되거나 **farm_admin**에서 관리. **일정관리 페이지**는 **작업 계획**(`farm_schedule_work_plans`)을 주 데이터로 조회·표시. |
| **API** | `GET/POST/PUT/DELETE /api/farms/:farmId/schedule-items` 이미 존재. POST/PUT/DELETE 는 “농장 직접 등록·수정·삭제”에 그대로 활용 가능. |
| **작업 계획·완료** | “예정 작업일자(시작·종료)”와 “완료 일자”를 담는 **별도 테이블·API·UI가 아직 없음**. (설계: 작업 계획 별도 테이블로 예정일+완료일 관리 예정.) |

---

## 2. 목표 정리

1. **농장 직접 일정 등록·수정·삭제**  
   - 농장이 대시보드(또는 farm_admin)에서 **자체 일정**을 추가·수정·삭제할 수 있게 한다.  
   - 기존 `farm_schedule_items` CRUD API를 활용한다.

2. **작업 계획·완료를 별도 테이블로 관리**  
   - **별도 테이블**에 **예정 작업일자**(시작일·종료일)와 **완료 일자**를 기록한다.  
   - 작업 계획은 **상황 이벤트 발생 시**(예: 전입 시 해당 방/칸 이동 등록 후 스케줄 기반 작업 목록을 즉시 별도 테이블에 저장) 또는 **사용자가 특정 날짜에 일정 추가 시**(추가한 작업 내용으로 다음 날짜 일정도 등록 가능) 생성된다.  
   - 그리드나 목록에서 **예정 구간**과 **완료 여부**를 표시·필터할 수 있게 한다.

## 3. 권한·역할 제안

기능별로 **누가** 할 수 있는지 아래처럼 구분하는 것을 제안한다.

| 기능 | 허용 역할 | 비고 |
|------|-----------|------|
| **일정 추가·수정·삭제** | **관리자**, **농장장**만 | 시스템 관리자·최고 관리자 + 해당 농장의 농장 관리자(farm_admin) 또는 소유자(owner). 일정 마스터를 바꾸는 기능이므로 제한. |
| **작업 계획 생성** | **이벤트 자동** 또는 **관리자·농장장·작업자** | 이벤트(전입 등) 발생 시 자동 생성; 사용자가 특정 날짜에 일정 추가 시 해당 내용으로 다음 날짜 작업 계획도 등록 가능. |
| **작업 완료 체크(완료/해제)** | **모든 작업자** | 해당 농장 접근 권한이 있는 사용자는 누구나 “이 작업 계획을 완료했다”고 체크·해제 가능. 예정·완료 모두 **작업 계획 별도 테이블**에 저장. |

**구현 시**  
- **일정 CRUD API** (`POST/PUT/DELETE .../schedule-items`): “관리자 또는 농장장”만 허용.  
- **작업 계획 API** (`GET/POST/PUT/DELETE .../schedule-work-plans`): 계획 조회·추가·수정·삭제; 완료 체크는 PATCH 등으로 해당 행의 완료일 설정. 해당 농장 접근 권한만 있으면 완료 체크 호출 가능.  
- **UI**: 일정 추가/수정/삭제는 관리자·농장장만; 작업 계획 보기·완료 체크(✓ 토글)는 권한 있는 모든 사용자에게 노출.

## 4. 단계별 진행 제안

### 4.1 단계 1: 농장 직접 일정 등록·수정·삭제 (CRUD)

**목표**  
농장이 **어디서** 일정을 추가/수정/삭제할지 정하고, 해당 화면에서 기존 API를 호출해 반영한다.

**방식 후보**

| 방식 | 장점 | 단점 | 비고 |
|------|------|------|------|
| **A. 대시보드 일정 페이지에서만** | 일정 조회와 등록/수정이 한 화면에서 이어짐. 그리드 셀·행 클릭으로 “이 시설/이 날에 일정 추가” 유도 가능. | 그리드 위에 폼/모달 설계 필요. 작업유형·기준·대상장소 등 입력 필드가 많음. | “일정(작업) 관리”를 대시보드에서 완결할 때 적합. |
| **B. farm_admin “농장 일정 관리”에서만** | 이미 농장 설정·시설·일정 템플릿을 다루는 화면이 있으면, 일정 목록/폼을 그쪽에 두기 단순. | 대시보드 일정 페이지는 계속 “조회 전용”이고, 등록하려면 farm_admin으로 이동해야 함. | 역할 분리(설정=admin, 실행/확인=대시보드)에 맞음. |
| **C. 둘 다 제공** | 대시보드에서 “빠른 추가” + farm_admin에서 “상세 목록·편집”. | 중복 UI·유지보수 부담. | 요구가 “어디서나 등록 가능”일 때. |

**권장**  
- **우선 B**로 진행: farm_admin에 “농장 일정 관리” 목록·추가/수정/삭제를 두고, 기존 `POST/PUT/DELETE /api/farms/:farmId/schedule-items` 를 사용.  
- 이후 “대시보드에서도 간단 등록”이 필요하면 **A**를 추가(예: 그리드 셀 클릭 → “이 시설·이 주에 일정 추가” 모달 → POST 후 그리드 갱신).

**구체 작업(단계 1)**  
1. **farm_admin** (또는 선택한 화면)에 “농장 일정” 목록 + 추가/수정/삭제 UI 구성.  
2. 작업유형(`schedule-task-types`), 기준(`schedule-basis-types`), 대상장소(`structureTemplateId`) 선택 및 dayMin/dayMax, description 등 입력.  
3. 추가 시 `POST /api/farms/:farmId/schedule-items`, 수정 시 `PUT .../schedule-items/:id`, 삭제 시 `DELETE .../schedule-items/:id` 호출.  
4. 대시보드 일정 페이지는 **GET** 만 사용하므로, farm_admin에서 변경한 내용이 그리드에 반영되도록 새로고침 또는 이후 단계에서 캐시 무효화만 고려.  
5. **작업 계획·완료**는 이 단계가 아니라 **별도 테이블**로 단계 2에서 다룬다(예정 시작/종료일 + 완료일).

---

### 4.2 단계 2: 작업 계획 **별도 테이블**(예정 시작/종료일 + 완료일) 및 API

**목표**  
“어느 일정 항목”에 대해 **예정 작업일자(시작일·종료일)**와 **완료 일자**를 **별도 테이블**에 저장·조회할 수 있게 한다. 작업 계획은 **상황 이벤트 발생 시 즉시** 생성되거나, 사용자가 일정을 추가할 때 생성된다.

**설계 방침: 작업 계획·완료는 별도 테이블로 관리**

- **farm_schedule_items**: “무슨 작업을, 어떤 조건(대상장소·기준·dayMin~dayMax)으로 할지” **정의**만 담당.  
- **작업 계획·완료**: **별도 테이블** 한 행에 **예정 시작일·예정 종료일**(며칠 계획인 경우 대비)과 **완료 일자**(완료 시 기록)를 함께 둔다.

**왜 별도 테이블인가**

| 구분 | farm_schedule_items | 작업 계획(별도 테이블) |
|------|---------------------|-------------------------|
| **역할** | “할 일” **정의** (템플릿·마스터) | “언제부터 언제까지 할 예정인지” + “언제 완료했는지” **실적** |
| **일자** | dayMin/dayMax 등 **조건**만 정의 | **예정 시작일·예정 종료일** + (선택) **완료일** |
| **행 수** | 일정 종류당 1행 | 같은 일정이라도 **계획 건마다 1행** (이벤트·사용자 추가 시 생성) |
| **생성 시점** | 관리자·농장장이 가끔 수정/삭제 | **이벤트 발생 시 즉시** 또는 **사용자 일정 추가 시** |

- **farm_schedule_items만** 쓰면: “이 일정을 3월 1일~3일 예정으로 잡고, 3월 2일에 완료했다”처럼 **예정 구간**과 **완료일**을 함께 쌓기 어렵다.  
- 따라서 **작업 계획 전용 테이블**을 두고, 1행 = 1건의 작업 계획(예정 시작·종료 + 완료일)으로 관리한다.

**작업 계획 생성 시점**

1. **이벤트 기반 생성**  
   - **상황 이벤트가 발생하면 바로** 해당 대상(방/칸 등)에 대한 작업 계획을 별도 테이블에 넣는다.  
   - 예: **전입** — 그 방(칸)에 전입하여 **이동 등록**을 하면, 그 시점에 해당 방/칸에 대해 스케줄(`farm_schedule_items`)에 의해 정해진 작업 일정 목록을 조회하고, **즉시 별도 테이블에 작업 계획 행**을 생성한다. 예정 시작일·종료일은 스케줄의 dayMin/dayMax·기준일 등으로 계산.
2. **사용자 기반 생성**  
   - 사용자가 **특정 날짜를 선택해 일정을 추가**하면, 그 추가한 작업 내용(일정 항목·대상 등)에 따라 **다음 날짜의 작업 계획**도 함께 등록할 수 있다. (예: “이 시설·이 일정을 다음 주까지 반복 등록” 등.)

**테이블 설계: `farm_schedule_work_plans`**

테이블명: **`farm_schedule_work_plans`** (작업 계획·완료 별도 테이블)

| 컬럼 | 타입 | NULL | 설명 |
|------|------|------|------|
| id | PK, auto increment | N | 고유 ID |
| farmId | UUID, FK → farms | N | 농장 (권한·파티션용) |
| farmScheduleItemId | INTEGER, FK → farm_schedule_items | N | 해당 일정 항목 |
| **taskTypeCategory** | **VARCHAR(50)** | **Y** | **작업 유형 대분류 (이동/환경 등).** `farm_schedule_task_types.category`와 동일한 값. 생성 시 해당 일정 항목(farm_schedule_items)의 작업유형(taskType)에서 category를 복사해 저장. "이동만", "환경만" 필터·그리드 표시 시 조회용. |
| roomId | UUID, FK | Y | 대상 방 (이벤트/대상 단위 시) |
| sectionId | UUID, FK | Y | 대상 칸 (이벤트/대상 단위 시) |
| plannedStartDate | DATEONLY | N | **예정 시작일** |
| plannedEndDate | DATEONLY | N | **예정 종료일** (며칠 계획이든 시작·끝 표현) |
| completedDate | DATEONLY | Y | **완료 일자** (완료 체크 시 기록) |
| completedAt | TIMESTAMP | Y | 완료 체크 시각 |
| completedBy | UUID, FK → users | Y | 완료 체크한 사용자 (감사·통계용) |
| createdAt, updatedAt | TIMESTAMP | N | 레코드 생성/수정 시각 |

- **taskTypeCategory**: 예) `move`(이동), `environment`(환경·시설), `vaccine`, `feed` 등 — `farm_schedule_task_types.category` 값과 동일. 작업 계획 생성 시 해당 `farm_schedule_item`의 `taskTypeId` → `farm_schedule_task_types.category`로 채운다.
- **인덱스**: `(farmId, plannedStartDate, plannedEndDate)`, `(farmScheduleItemId, plannedStartDate)`, `(roomId, sectionId)`, `(farmId, taskTypeCategory)` — 기간·일정·대상·**작업유형(이동/환경)**별 조회용.  
- **확장**: 발생 원인(이벤트 ID, 이동 ID 등)을 남기려면 `sourceType`, `sourceId` 같은 컬럼을 추가할 수 있음.

**API**

- `GET /api/farms/:farmId/schedule-work-plans?from=...&to=...&scheduleItemId=...&roomId=...&sectionId=...&taskTypeCategory=...`  
  → 기간·일정·대상·**작업유형(이동/환경 등)**별 작업 계획 목록. `taskTypeCategory`(예: move, environment)로 "이동만", "환경만" 필터 가능.  
- `POST /api/farms/:farmId/schedule-work-plans`  
  → body: `{ farmScheduleItemId, plannedStartDate, plannedEndDate, roomId?, sectionId? }`. **taskTypeCategory**는 서버에서 해당 farm_schedule_item의 taskType.category로 자동 채움.  
- `PATCH /api/farms/:farmId/schedule-work-plans/:id`  
  → body: `{ completedDate }` 또는 `{ completedDate: null }` — 완료 체크/해제.  
- `PUT /api/farms/:farmId/schedule-work-plans/:id`  
  → 예정 시작/종료일 수정.  
- `DELETE /api/farms/:farmId/schedule-work-plans/:id`  
  → 작업 계획 삭제.

### 4.3 단계 3: 작업 계획 표시·완료 체크 UI 및 그리드 반영

**목표**  
- 대시보드 일정 그리드에서 **별도 테이블**(`farm_schedule_work_plans`) 기준으로 **예정 구간(시작~종료)**과 **완료 여부**가 보이도록 하고,  
- 사용자가 “이 작업 계획을 완료했다”고 **체크(또는 해제)**할 수 있게 한다.

**방식**  
1. **그리드 표시**  
   - **작업 계획 별도 테이블**에서 기간·일정·대상별로 조회한 뒤, **예정 시작일·종료일**에 따라 해당 주/날짜 셀에 “작업 계획 있음” 표시.  
   - 해당 계획에 **completedDate**가 있으면 체크 아이콘(✓) 또는 배경색/배지로 “완료” 표시.  
   - 데이터는 `GET schedule-work-plans?from=주시작&to=주끝` 등으로 가져와, `plannedStartDate`~`plannedEndDate`·`scheduleItemId`·room/section 기준으로 그리드 셀과 매핑.

2. **완료 체크 UI**  
   - **옵션 1**: 그리드 셀(또는 셀 안 작업 계획 배지) 클릭 → 해당 작업 계획에 대해 완료 토글 (PATCH work-plan: completedDate 설정/해제).  
   - **옵션 2**: 작업 계획 목록/상세 모달에서 “완료” 체크.  
   - **옵션 3**: 둘 다 제공 (그리드에서 빠른 체크 + 모달에서 상세).

**구체 작업(단계 3)**  
1. 단계 2 설계대로 **별도 테이블** `farm_schedule_work_plans` 및 모델 추가.  
2. `GET/POST/PUT/PATCH/DELETE .../schedule-work-plans` API 구현.  
3. 대시보드 일정 페이지에서:  
   - 주(7일) 구간에 대해 **작업 계획** 조회.  
   - 셀 렌더 시 예정 구간(plannedStartDate~plannedEndDate)에 해당하는 계획 표시, 완료된 것은 ✓ 등으로 표시.  
4. 이벤트(전입 등) 발생 시 해당 방/칸에 대한 스케줄 기반 작업 계획 자동 생성 로직 연동.  
5. 사용자 일정 추가 시, 필요 시 다음 날짜 작업 계획도 등록하는 로직 연동.  
6. 셀(또는 작업 계획 배지) 클릭 시 완료 토글(PATCH) API 호출 후 그리드 갱신.

---

### 4.4 단계 4 (선택): 완료 필터·통계

- 그리드 또는 목록에 “전체 / 미완료만 / 완료만” 필터. (완료 여부는 **작업 계획 별도 테이블** `farm_schedule_work_plans`의 `completedDate` 기준.)  
- “이번 주 미완료 작업 계획” 요약 등 간단 통계.

## 5. 작업 추가 기능 구현 추천

**목표**: 사용자가 대시보드 일정 페이지에서 **작업 계획(farm_schedule_work_plans)** 을 수동으로 추가할 수 있게 한다.

### 5.1 백엔드: POST API 구현

- **필요**: `POST /api/farms/:farmId/schedule-work-plans` (현재 GET만 구현됨).
- **body**: `{ farmScheduleItemId, plannedStartDate, plannedEndDate, roomId?, sectionId? }`.
- **서버 처리**:
  1. farmScheduleItemId로 `FarmScheduleItem` 조회, 해당 항목의 `taskTypeId` → `FarmScheduleTaskType.category`를 읽어 **taskTypeCategory** 저장.
  2. roomId/sectionId는 해당 농장 소속인지 검증 후 저장.
  3. `FarmScheduleWorkPlan.create(...)` 후 201 + 생성된 행(include scheduleItem·taskType) 반환.
- **권한**: 문서상 작업 계획 생성은 “관리자·농장장·작업자” 가능. 기존 `checkFarmPermission`이 farm_admin/manager만 허용하면, **작업 추가용**은 동일 권한으로 두거나, “농장 접근 가능한 모든 역할”로 완화할지 정책 결정 후 적용.

### 5.2 UI 배치 추천: 대시보드 일정 페이지 + 모달

| 방식 | 설명 | 추천도 |
|------|------|--------|
| **A. 셀/날짜 클릭 → 모달** | 그리드의 **특정 셀(행+열)** 클릭 시 “이 날짜, 이 방·칸에 작업 추가” 모달 오픈. 행에서 roomId/sectionId, 열에서 날짜(plannedStartDate 기본값) 전달. | ✅ **추천** — 맥락(어디·언제)이 명확함. |
| **B. 상단 “작업 추가” 버튼** | 상단에 버튼 하나 두고 클릭 시 모달. 대상 방/칸·날짜는 모달 내에서 모두 선택. | 보조로 사용 가능(빠른 추가 후 대상 선택). |
| **C. 빈 칸 더블클릭** | 셀 더블클릭 시에만 추가 모달. 단일 클릭은 추후 “완료 토글” 등으로 예약. | A와 병행 시 입력 구분에 유리. |

**권장**: **A**를 기본으로 하고, 필요 시 **B**를 보조 버튼으로 추가.

### 5.3 모달 입력 항목

| 항목 | 필수 | 기본값/설명 |
|------|------|-------------|
| **일정(작업) 선택** | ✅ | `GET /api/farms/:farmId/schedule-items` 목록을 드롭다운으로 표시. 표시: 작업유형명 + 설명(또는 structureTemplate명). value = farmScheduleItemId. |
| **예정 시작일** | ✅ | 클릭한 셀의 날짜(또는 오늘). date 입력. |
| **예정 종료일** | ✅ | 예정 시작일과 동일(1일 작업) 또는 사용자 입력. date 입력. |
| **대상 방/칸** | 선택 | 셀 클릭 시 해당 행의 roomId, sectionId를 모달에 전달해 **읽기 전용**으로 표시. “상단 버튼”으로 열었을 때만 방/칸 선택 드롭다운 노출. |

- **roomId / sectionId**: 셀 클릭 시점의 행에서 가져와서 모달 hidden 또는 읽기 전용으로 넘기고, POST body에 포함.

### 5.4 프론트 플로우 요약

1. 그리드 **날짜 셀** 클릭 시: 해당 **행**의 roomId/sectionId(또는 roomIds)와 클릭한 **열**의 날짜(plannedStartDate) 수집.
2. **“작업 추가” 모달** 오픈. (기존 일정 페이지에 모달 HTML 추가 또는 동적 생성.)
3. 모달에서 **일정 항목(farmScheduleItemId)** 드롭다운 로드: `GET .../schedule-items` 호출 후 옵션 렌더.
4. **예정 시작일·종료일** 기본값: 클릭한 날짜. 필요 시 수정 가능.
5. **저장** 클릭 시 `POST /api/farms/:farmId/schedule-work-plans` 호출. body: `{ farmScheduleItemId, plannedStartDate, plannedEndDate, roomId?, sectionId? }`.
6. 성공 시 모달 닫고, **해당 주 작업 계획 재조회** 후 그리드 갱신(`loadWorkPlansForCurrentWeekAndRender` 등).

### 5.5 예외·검증

- **일정 항목이 0개**인 농장: 드롭다운 비어 있음. “먼저 farm_admin에서 일정을 등록하세요” 안내 문구 표시.
- **roomId/sectionId**: 해당 농장 소속 여부는 서버에서 검증. 프론트는 “셀 클릭 시 해당 행 정보”만 넘기면 됨.
- **날짜**: plannedStartDate ≤ plannedEndDate 검증은 서버에서 수행 권장.

### 5.6 구현 순서 제안

1. **POST** `.../schedule-work-plans` API 구현 (taskTypeCategory 자동 채움, roomId/sectionId 검증).
2. 대시보드 일정 페이지에 **“작업 추가” 모달** 마크업 추가 (일정 선택, 예정 시작/종료일, 대상 표시/선택).
3. 그리드 **셀 클릭** 시 해당 (행, 열) 정보로 모달 열기 + 일정 목록 로드.
4. 모달 **저장** 시 POST 호출 후 주 단위 재조회·그리드 갱신.

이 순서로 진행하면 “작업 추가” 기능을 문서 설계에 맞게 도입할 수 있습니다.

## 7. 설계 시 참고 사항

- **일정(정의) vs 작업 계획·완료: 별도 테이블**  
  - **일정 정의**: `farm_schedule_items` — “무슨 작업을, 어떤 조건으로 할지”만 저장.  
  - **작업 계획·완료**: **별도 테이블** `farm_schedule_work_plans` — “예정 작업일자(시작일·종료일)”와 “완료 일자”를 1행에 기록.  
  - 그리드는 “이 시설 유형에 정의된 일정 목록”과 **별도 테이블**의 작업 계획(예정 구간·완료 여부)을 함께 보여준다.

- **작업 계획 생성 시점**  
  - **이벤트 기반**: 전입 등 이동 등록 시 해당 방/칸에 대해 스케줄 기반 작업 목록을 조회해 **즉시** 작업 계획 행 생성.  
  - **사용자 기반**: 사용자가 특정 날짜에 일정을 추가할 때, 그 내용으로 **다음 날짜 작업 계획**도 등록 가능하도록 구현 검토.

- **farm_admin vs 대시보드**  
  - **일정 마스터(추가/수정/삭제)**: farm_admin “농장 일정 관리”에서. **관리자·농장장만** 허용(위 §3 참고).  
  - **작업 계획 보기·완료 체크**: 대시보드 **일정(작업) 관리 페이지** 그리드에서. 그리드 표시 데이터는 **`farm_schedule_work_plans`**에서 불러오며, 예정·완료는 이 테이블 기준. **모든 작업자**가 완료 체크 가능(위 §3 참고). 페이지 작업 정의는 `docs/dashboard_schedule_page_work.md` 참고.

- **권한 요약**  
  - **일정 추가·수정·삭제**: 관리자, 농장장만.  
  - **작업 계획 생성**: 이벤트 자동 또는 관리자·농장장·작업자(일정 추가 시 다음 날짜 등록 등).  
  - **작업 완료 체크**: 해당 농장 접근 권한이 있는 모든 사용자. (저장 위치: **작업 계획 별도 테이블** `farm_schedule_work_plans`의 completedDate 등.)

이 문서를 기준으로 **1 → 2 → 3 → 4** 순으로 진행하면, “농장 직접 일정 등록”과 “**작업 계획 별도 테이블**(예정 시작/종료일 + 완료일)을 이용한 작업 계획·완료 관리”를 단계적으로 도입할 수 있으며, **작업 추가**(§5) 구현 시 위 추천(셀 클릭 → 모달)을 따르면 된다.
