# 예정 작업 등록 화면 흐름 설계

> **시나리오**: 환경설정의 일정 마스터에서 작업을 등록한 뒤, 일정 관리 화면에서 해당 작업을 예정으로 등록하는 흐름을 정의합니다.

---

## 1. 시나리오 개요

| 구분 | 설명 |
|------|------|
| **목표** | 사용자가 "어떤 작업을, 언제, 어느 칸에서 할지" 예정으로 등록할 수 있게 함 |
| **전제** | 예정으로 등록할 **작업(작업 계획)** 은 먼저 **환경설정 > 일정 마스터**에서 정의되어 있어야 함 |
| **흐름** | ① 일정 마스터에서 작업(기초 일정) 등록 → ② 일정 관리 캘린더에서 칸·날짜 선택 → ③ 예정 등록 모달에서 해당 작업 선택 후 등록 |
| **단계별 선택** | 일정 마스터는 **시설 → 구분 → 작업유형 → 기준** 순의 단계별 선택. 예정 등록 모달은 **현재 단계별 선택이 아님**(작업 계획 단일 드롭다운). 동일 흐름으로 보완 시 §3.3 참고. |

즉, **작업 정의(일정 마스터)** 와 **작업 예정 등록(일정 관리)** 이 한 시나리오로 이어지도록 설계합니다.

---

## 2. 화면 흐름도 (전체)

```
[대시보드]
    │
    ├─► [환경설정] ─► [일정 마스터] ─► 기초 일정(작업) 등록
    │                      │
    │                      │  사육시설/일반시설 → 시설 선택
    │                      │  → 구분 → 작업유형 → 기준 → 기준내용·작업내용
    │                      │  → 「기초 일정 저장」 → 작업 목록에 추가
    │                      ▼
    │                 schedule_work_plans (작업 계획 마스터)
    │
    └─► [일정 관리] ─► 주간 캘린더
                           │
                           │  칸·날짜의 칸 클릭
                           ▼
                      「예정 작업 입력」 모달 (바로 열림)
                           │   · 칸 / 날짜: 자동 표시
                           │   · 작업 계획(드롭다운) ← 일정 마스터에서 등록한 작업 목록
                           │   · 실행 유형(분만/이동/시설·점검)
                           │   · 메모
                           │   · [등록]
                           │        │
                           ▼        ▼
                      schedule_executions (pending) 생성 → 해당 칸에 예정 표시
```

---

## 3. 단계별 화면 흐름

### 3.1 1단계: 환경설정 > 일정 마스터에서 작업 등록

**진입 경로**

- 대시보드 → **설정(톱니바퀴 등)** → 좌측 메뉴 **일정 마스터**  
- URL: `/dashboard/settings/schedule-masters`

**화면 구성 (기초 일정 관리)**

| 영역 | 내용 |
|------|------|
| 상단 | 제목 「기초 일정 관리」, 설명 문구 |
| 농장시설 | **사육시설** / **일반시설** 탭, 시설 목록에서 **시설 1개 선택** |
| 구분 | 선택한 시설에 대한 구분 목록. 단일 선택. 필요 시 「구분 목록 선택」으로 정의 추가 |
| 작업유형 | 선택한 구분에 대한 작업유형 목록. 단일 선택. 「작업유형 목록 선택」으로 추가 가능 |
| 기준 | 선택한 작업유형에 대한 기준 목록. 단일 선택. 「기준 목록 선택」으로 추가 가능 |
| 기준내용 | 기준 유형(일, 요일, 기간 등)에 따른 입력 + **기초 일정 저장** 버튼 |
| 작업내용 | 작업 상세 내용 입력 + **이동 대상 시설**(작업유형이 **이동**일 때만 표시, 사육시설 중 선택) + 동일한 **기초 일정 저장** 버튼 |
| 하단 | **작업 목록 (schedule_work_plans)** — 시설·구분·작업유형·기준·기준내용·작업내용·**이동 대상**(이동인 경우) 열로 표시, 수정·삭제·순서 변경 가능 |

**사용자 시나리오 (작업 등록)**

1. 농장시설에서 **사육시설** 또는 **일반시설** 선택 후, 사용할 **시설 1개** 선택.
2. **구분**에서 원하는 구분 선택 (없으면 구분 정의 후 목록에서 선택).
3. **작업유형**에서 원하는 작업유형 선택 (예: 분만, 이동, 시설·점검 등).
4. **기준**에서 원하는 기준 선택 (예: 매일, 특정 요일, 기간 등).
5. **작업유형이 "이동"인 경우** **이동 대상 시설**에서 어느 사육시설로 이동하는 작업인지 선택 (필수).
6. **기준내용** 입력 후 **기초 일정 저장** 클릭  
   또는 **작업내용** 입력 후 **기초 일정 저장** 클릭.  
   → 한 행이 `schedule_work_plans`(농장 스코프 시 `farm_id` 포함)에 저장됨. 이동 작업은 `target_structure_template_id`(이동 대상 시설)도 함께 저장됨.
7. 하단 **작업 목록**에서 저장된 항목 확인. 필요 시 수정·삭제·순서 변경.

**결과**

- 해당 농장에서 **일정 관리** 화면의 「작업 계획」 드롭다운에 노출될 **작업(기초 일정)** 이 마스터에 등록됨.
- 일정 관리 페이지는 `getFarmScheduleWorkPlansMaster(farmId)` 등으로 이 목록을 조회해 예정 등록·바로 완료 시 사용.
- **이동** 작업은 `schedule_work_plans.target_structure_template_id`(이동 대상 시설)를 함께 저장하며, 완료 처리 시 도착 칸 제안 등에 활용 가능. (상세 설계: `일정마스터_이동_대상시설_추천.md` 참고.)

### 3.2 2단계: 일정 관리에서 예정 등록

**진입 경로**

- 대시보드 → **일정 관리**  
- URL: `/dashboard/schedule`

**화면 구성**

- 상단: 주간 이동, 캘린더 열기 등.
- 본문: **농장 구조(동·사육시설·방·칸)** × **요일** 그리드.  
  각 칸을 누르면 **예정 작업 입력** 모달이 열림. 칸에는 해당 칸·날짜의 **예정(pending)**·완료 목록이 표시됨.

**사용자 시나리오 (예정 등록)**

1. **칸·날짜**의 **칸**을 클릭 → **「예정 작업 입력」** 모달이 바로 열림.
2. 모달에서 **일정 마스터와 동일한 단계별 선택**:
   - **칸 / 날짜**: 자동 표시 (클릭한 칸·날짜). 해당 칸이 속한 시설은 자동 결정됨.
   - **구분** → **작업유형** → **기준** → **작업 계획**: 순서대로 드롭다운 선택 (해당 칸·시설에 연결된 작업만 노출).
   - **실행 유형**: 분만 / 이동 / 시설·점검 (선택에 따라 완료 처리 시 입력 항목이 달라짐).
   - **메모**: 선택 입력.
3. **등록** 클릭 → `createFarmScheduleExecution` 호출, `schedule_executions`에 **pending** 1건 생성.
4. 캘린더 해당 칸에 방금 등록한 예정 항목이 표시됨.

**연결 관계**

- **작업 계획** 드롭다운 옵션 = 1단계 일정 마스터에서 저장한 `schedule_work_plans` (현재 농장 + 전역).
- 예정 등록 시 전송: `workPlanId`, `sectionId`, `executionType`, `scheduledDate`, `memo` 등.
- **이동 작업**의 경우 작업 계획에 **target_structure_template_id**(이동 대상 시설)가 저장되어 있으므로, **완료 처리** 시 "도착 칸" 드롭다운의 기본값 또는 제안으로 해당 시설에 속한 칸만 노출·우선 제안하는 등 UX 보완에 활용할 수 있음.

**단계별 선택 흐름 여부**

- **일정 마스터(3.1)** 는 **단계별 선택** 구조입니다.  
  시설 → 구분 → 작업유형 → 기준 → (기준내용/작업내용) 순으로 선택하며, 이전 단계 선택에 따라 다음 옵션이 필터링됩니다.
- **예정 등록 모달(3.2)** 도 **일정 마스터와 동일한 단계별 선택** 구조로 구현됩니다.  
  칸·날짜(및 해당 칸이 속한 시설)는 이미 확정된 상태에서, **구분 → 작업유형 → 기준 → 작업 계획** 순으로 드롭다운을 선택하고, 실행 유형·메모를 입력한 뒤 등록합니다.  
  일정 마스터 페이지로 이동하지 않고 모달 내에서 단계별로 선택할 수 있습니다.

### 3.3 예정 등록을 일정 마스터와 동일한 단계별 선택 흐름으로 보완(선택)

일정 마스터와 동일한 UX로 예정 등록도 **단계별 선택**으로 설계하려면 아래와 같이 바꿀 수 있습니다.

| 단계 | 화면/입력 | 비고 |
|------|-----------|------|
| 0 | **칸·날짜** | 이미 확정 (빈 영역 클릭 시점). 칸이 속한 **시설(템플릿)** 은 자동 결정. |
| 1 | **구분** | 해당 시설에 대한 구분 목록만 노출. 단일 선택. |
| 2 | **작업유형** | 선택한 구분에 대한 작업유형 목록만 노출. 단일 선택. |
| 3 | **기준** | 선택한 작업유형에 대한 기준 목록만 노출. 단일 선택. |
| 4 | **작업 계획** | 위에서 선택한 “시설·구분·작업유형·기준” 조합에 해당하는 `schedule_work_plans`만 드롭다운에 노출. 하나 선택. |
| 5 | **실행 유형·메모** | 기존과 동일. [등록] 클릭. |

**효과**

- 일정 마스터에서 작업 등록할 때와 같은 **시설 → 구분 → 작업유형 → 기준 → 작업** 순서를 유지해, 사용자 인지 부담을 줄일 수 있음.
- 작업 계획 목록이 단계별로 좁혀지므로, **해당 칸·시설에 맞는 작업만** 골라 예정 등록하기 쉬움.

**구현 시 참고**

- 일정 마스터와 동일한 API로 구분/작업유형/기준 목록 조회 (시설 = 칸의 `structureTemplateId` 등으로 결정).
- 최종 단계에서 “시설·구분·작업유형·기준” 조합에 맞는 work_plan 목록만 필터링해 드롭다운에 표시.

## 4. 데이터 흐름 요약

| 단계 | 화면 | 사용자 행동 | API/데이터 |
|------|------|------------|------------|
| 1 | 환경설정 > 일정 마스터 | 시설·구분·작업유형·기준·(이동일 때 이동 대상 시설)·기준내용/작업내용 선택 후 「기초 일정 저장」 | `createFarmScheduleWorkPlanMaster` 등 → `schedule_work_plans` 저장 (이동 시 `target_structure_template_id` 포함) |
| 1 | 환경설정 > 일정 마스터 | 작업 목록 조회·수정·삭제 | `getFarmScheduleWorkPlansMaster`, `updateFarmScheduleWorkPlanMaster`, `deleteFarmScheduleWorkPlanMaster` 등 |
| 2 | 일정 관리 | 칸 클릭 → 「예정 작업 입력」 모달 바로 열림 → 등록 | — |
| 2 | 일정 관리 | 작업 계획 선택 후 「등록」 | `getFarmScheduleWorkPlansMaster`로 옵션 구성, `createFarmScheduleExecution` → `schedule_executions` (status=pending) |

## 5. 정리

- **예정 작업 등록**은 **환경설정의 일정 마스터에서 작업을 먼저 등록하는 시나리오**를 전제로 설계합니다.
- 1단계: **일정 마스터**에서 시설·구분·작업유형·기준·기준내용/작업내용을 정하고 「기초 일정 저장」으로 작업(기초 일정)을 등록합니다.
- 2단계: **일정 관리** 캘린더에서 칸·날짜의 **칸**을 누르면 **「예정 작업 입력」** 모달이 바로 열리고, 위에서 등록한 **작업 계획**을 골라 예정을 등록합니다.
- 이렇게 하면 “작업 정의(일정 마스터)”와 “작업 예정 등록(일정 관리)”이 한 흐름으로 이어집니다.

## 6. 예정 등록 이후 흐름 (작업 클릭)

캘린더 칸에 이미 **예정(pending)** 이 있을 때, 해당 **작업을 클릭**하면 다음 선택 모달이 뜹니다.

| 선택 | 동작 |
|------|------|
| **완료 처리** | 해당 예정 건에 대한 완료 처리 모달 열림. 분만/이동 등 실행 유형에 따라 마리수·도착 칸·이벤트 유형 등을 입력 후 완료 → `schedule_executions` status=completed, 결과 데이터 생성. |
| **수정** | 예정 건의 날짜·메모 등 수정 (기능 준비 중 시 placeholder 모달만 노출). |

**흐름 요약**

```
일정 관리 칸에서 [예정 항목] 클릭
    → 「작업 선택」 모달 (완료 처리 / 수정)
        → [완료 처리] → 완료 처리 모달 → API 호출 → 완료
        → [수정]     → 예정 수정 모달 (준비 중이면 안내만)
```

## 7. 바로 완료 흐름 요약

**진입**: (현재 구현은 예정건 클릭 후 완료 처리 등 다른 경로. 빈 칸에서의 바로 완료 진입은 필요 시 별도 버튼/메뉴로 추가 가능.)

**특징**

- 예정 건을 만들지 않고, **작업 계획 + 실행 유형 + 작업일·칸**을 선택한 뒤 곧바로 완료 API 호출.
- 분만이면 `directCompleteFarmScheduleExecutionBirth`, 이동이면 `directCompleteFarmScheduleExecutionMove` 등 사용.
- 일정 마스터에 등록된 **작업 계획** 목록은 예정 등록과 동일하게 `getFarmScheduleWorkPlansMaster`로 조회.

## 8. 예외·엣지 케이스

| 상황 | 권장 처리 |
|------|-----------|
| **농장 미선택** | 환경설정 > 일정 마스터 진입 시 "선택된 농장이 없습니다. 먼저 농장을 선택해 주세요." 안내. 일정 관리도 농장 선택 전이면 동일하게 안내. |
| **작업 계획 0건** | 예정 등록·바로 완료 모달에서 작업 계획 드롭다운이 비어 있음. "등록할 작업이 없습니다. 환경설정 > 일정 마스터에서 먼저 작업을 등록하세요." 문구 + 일정 마스터 링크(선택) 노출. |
| **작업 계획 선택 안 함** | 「등록」/「완료 처리」 시 "작업 계획을 선택해 주세요." 검증 메시지. |
| **등록/완료 API 실패** | 모달 내 에러 영역에 메시지 표시, 모달은 유지해 재시도 가능하게 함. |
| **일정 마스터에서 작업 삭제** | 이미 예정으로 등록된 실행 건의 work_plan_id는 유지. 목록 표시 시 삭제된 작업은 "삭제된 작업" 등으로 fallback 표시하거나, 조회 시 전역+농장 병합 정책에 따라 처리. |

---

## 9. 작업 계획 노출 범위 (칸·시설 연동)

- **현재**: 예정 등록·바로 완료 시 `getFarmScheduleWorkPlansMaster(farmId)`로 전체 작업 목록을 사용.
- **선택 보완**: 해당 **칸(section)** 이 속한 **사육시설(템플릿)** 에 연결된 작업만 드롭다운에 노출할 수 있음.  
  - 일정 마스터의 작업은 "시설(구조 템플릿)"과 연결되므로, 칸의 `structureTemplateId`와 work_plan의 시설 정보를 매칭해 필터링하면 됨.
- 이렇게 하면 "이 칸에는 어떤 작업만 예정으로 넣을 수 있다"는 규칙이 명확해짐.

## 12. 예정 작업 입력 모달 구조 정리 (효율화 설계)

### 12.1 현재 모달 필드

| 필드 | 역할 | 비고 |
|------|------|------|
| 대상 | 칸(건물/사/방/칸) | 읽기 전용, 클릭한 칸 |
| 날짜 | 예정일 | 읽기 전용 |
| 구분 | 일정 마스터의 구분 선택 | 작업 계획 필터 1단계 |
| 작업유형 | 일정 마스터의 작업유형 선택 | 작업 계획 필터 2단계 |
| 기준 | 일정 마스터의 기준 선택 | 작업 계획 필터 3단계 |
| 작업 계획 | 최종 선택한 schedule_work_plans 1건 | 구분·작업유형·기준에 따라 목록 필터됨 |
| 실행 유형 | 분만/이동/시설·점검 | 완료 처리 시 입력 분기용 |
| 메모 | 선택 입력 | |

### 12.2 중복·불필요 요소 정리

- **라벨 이중 표시**: 구분/작업유형을 "섹션 제목"과 "행 라벨"에 둘 다 쓰면 중복. → **섹션 제목 제거**, 각 행만 라벨(구분, 작업유형, 기준, 작업 계획, 실행 유형, 메모)로 표시해 하나로 통일.
- **데이터 관점**  
  - **구분·작업유형·기준**은 "작업 계획" 후보를 좁히기 위한 **필터**일 뿐이며, 최종 저장되는 것은 **작업 계획 ID(work_plan_id)** 하나.  
  - **실행 유형**은 작업 계획의 작업유형(분만/이동/시설·점검)에서 **자동 추론 가능**하나, 사용자가 다른 유형으로 바꿀 수 있도록 드롭다운을 두는 경우도 있음.

### 12.3 효율적 설계 옵션

| 방안 | 모달 필드 | 장점 | 단점 |
|------|-----------|------|------|
| **A. 최소 필드** | 대상, 날짜, **작업 계획**(단일 드롭다운), 메모 | 입력 단계 최소. 실행 유형은 작업 계획에서 자동 설정. | 작업 계획 옵션이 많으면 드롭다운이 길어짐. |
| **B. 단계별 필터(현행)** | 대상, 날짜, 구분 → 작업유형 → 기준 → 작업 계획, (실행 유형), 메모 | 해당 칸·시설에 맞는 작업만 단계적으로 좁혀서 선택 가능. | 필드 수 많음. 실행 유형은 작업 계획과 일부 중복. |
| **C. 절충** | 대상, 날짜, **작업 계획**(드롭다운, 옵션 텍스트에 "구분 / 작업유형 / 기준 - 작업내용" 표기), 메모. 실행 유형은 기본값만 자동, 필요 시에만 노출. | 한 줄로 고르되, 옵션만으로 구분·작업유형·기준을 구분 가능. 실행 유형은 필요 시만. | 작업 계획 목록이 매우 많으면 드롭다운이 여전히 김. |

### 12.4 권장

- **UI**: 라벨 이중 표시 제거(섹션 제목 없이 행 라벨만 사용) → 적용 완료.
- **구조**:  
  - **당장**: 현행 유지(구분 → 작업유형 → 기준 → 작업 계획 → 실행 유형 → 메모).  
  - **추후 단순화 시**: 방안 C 적용 검토 — "작업 계획" 한 필드로 통합하고, 옵션 라벨에 구분/작업유형/기준을 포함. 실행 유형은 기본값 자동 + 필요 시에만 수정 노출.
