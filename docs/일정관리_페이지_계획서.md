# 일정 관리 페이지 계획서

> 현재 프로젝트(양돈 농장 관리 시스템)의 일정 관리 기능을 정리한 계획·참고 문서입니다.  
> Go API + Next.js 기준으로, 코드·DB·기존 문서를 바탕으로 작성했습니다.

---

## 1. 개요

### 1.1 목적

- **전역 기초 일정**: 시스템 관리자가 “어떤 시설·구분·기준·작업유형 조합으로 일정을 쓸지” 템플릿을 정의.
- **농장 일정 실행**: `schedule_executions` 기반으로 예정 등록·분만/이동 완료 API 연동 완료. (계획서 §3, §4, §5)
- **일정 관리 페이지(`/farm/schedule`)**: 주간 캘린더 + 농장구조(건물→동→방→칸) 테이블에서 칸×날짜별 예정/완료 표시, 예정 등록·바로 완료·완료 처리 모달 연동. 상세 분석은 §7 참고.

### 1.2 범위

| 구분 | 설명 | 권한 |
|------|------|------|
| **기초 일정 관리** | 전역 `schedule_work_plans` + 구분/기준/작업유형 정의·목록 | 시스템 관리자(Super Admin) |
| **농장 일정** | `schedule_executions` 예정/완료, 분만·이동 완료 API | 농장 관리자(farm_admin/manager) |
| **일정 관리 페이지** | `/farm/schedule` — 주간 캘린더, 예정 등록·완료·바로 완료 (§7) | 해당 농장 접근 권한 사용자 |

## 2. 현재 구조 요약 (2단계)

```
[전역 - Admin]
  structure_templates (대상장소/사육시설)
       ↓
  schedule_sortation_definitions (구분 정의)
  schedule_sortations (시설별 구분 목록)
       ↓
  schedule_jobtype_definitions (작업유형 정의)
  schedule_jobtypes (구분별 작업유형 목록)
       ↓
  schedule_criteria_definitions (기준 정의)
  schedule_criterias (작업유형별 기준 목록)
       ↓
  schedule_work_plans (기초 일정 1행 = 시설+구분+작업유형+기준+criteria_content, work_content)



### 2.0 통합 운영 방식 (현재 적용)

초기의 `f_schedule_*`(농장 복제 테이블) 방식은 제거했고, 현재는 **단일 전역 테이블 `schedule_*`** 에서
`farmId` 컬럼으로 전역/농장 데이터를 구분해 운영합니다.

- `farmId IS NULL` → 전역(시스템 공통) 데이터
- `farmId = farms.id` → 해당 농장 전용 데이터
- `is_deleted = true` → 화면상 삭제(소프트 삭제)

#### 2.0.1 권한/삭제 정책

- **전역 데이터 삭제 권한**: 시스템 관리자만 가능 (Admin API)
- **농장 사용자**: 전역 데이터 삭제 불가
- 모든 삭제는 물리 삭제 대신 `is_deleted=true` 처리

#### 2.0.2 조회 정책

- Admin 화면: `farmId IS NULL AND is_deleted=false` 데이터만 조회
- Farm 화면: `(farmId IS NULL OR farmId=<현재 농장>) AND is_deleted=false` 조회

## 3. schedule_executions 확정안 (선행 고정)

돈군/이동/재고 연계 구현 전에, 일정 실행 단위를 `schedule_executions`로 먼저 고정합니다.  
핵심은 **계획(schedule_work_plans)과 실행(schedule_executions)을 분리**하는 것입니다.

> **요약 박스: 실행 파이프라인(고정)**
> 1. `schedule_work_plans`(계획) 기준으로 실행건 생성  
> 2. 생성된 실행건은 `schedule_executions.status='pending'`(예정)으로 저장  
> 3. 일정관리 화면은 `pending` 실행건을 캘린더에 표시  
>    - 표시 단위는 **칸(section)** 이며, `section_id + scheduled_date`가 일치하는 셀에만 노출  
> 4. 작업자가 분만/이동 완료 처리 API 호출  
> 5. 도메인 반영(돈군/원장/현재고) 성공 후 같은 실행건을 `completed`로 전이  
> 6. `result_ref_type/result_ref_id`로 완료 결과(분만/이동 이벤트) 연결
>
> ※ 예정/완료는 물리 테이블 분리가 아니라 `schedule_executions.status`로 관리

### 3.1 왜 먼저 고정해야 하는가

- 동일 일정의 중복 완료 방지(idempotency)
- 완료 API(`complete-birth`, `complete-move`)의 기준 키 통일
- 분만/이동 완료 결과를 돈군/재고 테이블과 안정적으로 연결

### 3.2 경로(라우트) 확정

- `POST /api/farms/:farmId/schedule-executions` (캘린더에서 예정 등록)
- `POST /api/farms/:farmId/schedule-executions/:executionId/complete-birth`
- `POST /api/farms/:farmId/schedule-executions/:executionId/complete-move`

> `executionId`는 "해당 농장/칸/날짜/작업 1건의 실행 단위"를 의미한다.

### 3.3 테이블 스키마(초안)

#### 3.3.1 schedule_executions

| 컬럼 | 타입 | 비고 |
|------|------|------|
| id | UUID | PK |
| farm_id | UUID | NOT NULL, FK → farms.id |
| work_plan_id | INTEGER | NOT NULL, FK → schedule_work_plans.id |
| section_id | UUID | NULL, FK → farm_sections.id |
| execution_type | VARCHAR(20) | NOT NULL (`birth`,`move`,`inspection`...) |
| scheduled_date | DATE | NOT NULL |
| status | VARCHAR(20) | NOT NULL DEFAULT `pending` (`pending`,`completed`,`skipped`,`cancelled`) |
| completed_at | TIMESTAMPTZ | NULL |
| completed_by | UUID | NULL, FK → users.id |
| result_ref_type | VARCHAR(30) | NULL (`farrowing_event`,`movement_event`,`shipment_event`...) |
| result_ref_id | UUID | NULL |
| idempotency_key | VARCHAR(80) | NULL (farm 기준 unique) |
| memo | TEXT | NULL |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT now() |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT now() |

권장 제약:
- unique: `(farm_id, idempotency_key)` where `idempotency_key IS NOT NULL`
- index: `(farm_id, scheduled_date, status)`

#### 3.3.2 컬럼 상세 설명

| 컬럼 | 역할 | 상세 설명 | 운영 메모 |
|------|------|-----------|-----------|
| `id` | 실행건 PK | 실행 단위 1건의 고유 식별자(UUID) | API 경로의 `executionId`로 사용 |
| `farm_id` | 테넌트 키 | 농장 단위 데이터 격리를 위한 필수 키 | 모든 조회/수정 쿼리에 강제 포함 |
| `work_plan_id` | 계획 참조 | 실행건이 어떤 계획(`schedule_work_plans`)에서 생성됐는지 연결 | 계획 변경 이력 추적 기준 |
| `section_id` | 위치 참조 | 실행 대상 칸(위치). 위치 미지정 작업은 NULL 가능 | 위치 기반 캘린더 표시/필터링 키 |
| `execution_type` | 실행 분류 | 실행 종류(`birth`,`move`,`inspection` 등) | 완료 API 분기 기준 |
| `scheduled_date` | 예정일 | 실행 예정 날짜 | 캘린더 주간/일간 조회 핵심 조건 |
| `status` | 상태 전이 | `pending`,`completed`,`skipped`,`cancelled` | 예정/완료 물리 테이블 분리 없이 상태로 관리 |
| `completed_at` | 완료 시각 | 완료 처리된 시점 | `status='completed'`일 때 필수 |
| `completed_by` | 완료 처리자 | 완료 처리한 사용자 ID | 사용자 감사 로그 기준 |
| `result_ref_type` | 결과 타입 | 완료 결과가 연결된 도메인 이벤트 타입 | 예: `farrowing_event`,`movement_event` |
| `result_ref_id` | 결과 ID | 완료 결과 이벤트의 PK(UUID) | `result_ref_type`와 쌍으로 사용 |
| `idempotency_key` | 멱등 키 | 중복 완료/재시도 안전 처리를 위한 키 | farm 기준 partial unique 적용 |
| `memo` | 비고 | 작업자 메모/특이사항 | UI 입력값 그대로 저장 가능 |
| `created_at` | 생성 시각 | 실행건 생성 시각 | 기본값 `now()` |
| `updated_at` | 수정 시각 | 실행건 변경 시각 | 상태 전이 시 갱신 |

#### 3.3.3 구조(관계) 요약

- `schedule_work_plans`(계획) 1건에서 `schedule_executions`(실행) N건이 생성됨
- 실행건은 기본 `status='pending'`으로 생성되고, 완료 API 성공 시 `completed`로 전이됨
- 완료 시 도메인 이벤트를 생성하고 `result_ref_type/result_ref_id`로 연결함
  - 분만 완료: `farrowing_events`
  - 이동 완료: `pig_movement_events`
- `section_id`, `scheduled_date`, `status` 조합으로 캘린더/작업 목록 조회 성능을 확보함
- **실행/완료의 위치 단위는 항상 칸(section)** 으로 고정한다.
- 방(room) 레벨 텍스트는 구조 안내(요약)로만 취급하며, 날짜별 실행건으로 취급하지 않는다.

> **참고**: 일정 마스터(반복)와 일정 관리(날짜 선택·1회성)의 칼럼 구분은 `일정_칼럼_구조_정리.md`를 참고하세요.  
> 1회성 등록 시 `work_plan_id` NULL 허용, `sortation_id`, `jobtype_id`, `work_content` 직접 저장 구조로 확장 예정.

#### 3.3.4 무결성 체크(권장)

- `status='completed'`이면 `completed_at`, `completed_by`는 NULL 불가
- `result_ref_type`와 `result_ref_id`는 둘 다 NULL이거나 둘 다 값 존재
- `idempotency_key`는 같은 `farm_id` 내 중복 금지(partial unique)
- `execution_type`, `status`는 허용값 체크 제약(ENUM 또는 CHECK) 권장

### 3.4 상태 전이 규칙(고정)

- 기본: `pending`
- 완료 API 성공 시: `pending -> completed`
- 이미 `completed`면:
  - 동일 idempotency_key 재호출: 200(같은 결과 반환)
  - 다른 요청: 409(중복 완료)
- `cancelled`, `skipped` 상태는 완료 API 대상 아님

### 3.5 완료 결과 연결 규칙

#### 3.5.1 분만 완료
- `result_ref_type = 'farrowing_event'`
- `result_ref_id = farrowing_events.id`
- 함께 처리:
  - 돈군 생성(`pig_groups`)
  - 칸 재고 원장 IN(`section_inventory_ledger`)
  - 필요 시 후속 실행건 자동 생성

#### 3.5.2 이동 완료
- `result_ref_type = 'movement_event'`
- `result_ref_id = pig_movement_events.id`
- 함께 처리:
  - 이동 이벤트/라인 생성
  - 칸 재고 원장 OUT/IN 반영
  - 돈군 두수/위치/상태 갱신

### 3.7 트랜잭션 단위(필수)

완료 API는 아래를 **한 트랜잭션**으로 처리한다.

1. `schedule_executions` 행 `FOR UPDATE` 잠금
2. 상태/중복 검증
3. 도메인 데이터 반영(돈군/이동/원장)
4. `result_ref_type/id` 기록
5. `status=completed`, `completed_at/by` 업데이트

실패 시 전체 롤백하여 정합성을 보장한다.

### 3.8 캘린더 직접 등록/완료 (공통 UX)

#### 3.8.1 예정 등록

1. 작업자가 캘린더에서 날짜 + 칸 선택
2. "예정 등록" 액션 클릭
3. 작업유형/계획(`work_plan_id`)/메모 입력
4. `POST /api/farms/:farmId/schedule-executions` 호출
5. `status='pending'` 실행건 생성 후 해당 셀에 즉시 표시

#### 3.8.2 완료 등록

1. 예정건 클릭 후 작업유형별 완료 모달 입력
2. 완료 API 호출
   - 분만: `POST /api/farms/:farmId/schedule-executions/:executionId/complete-birth`
   - 이동: `POST /api/farms/:farmId/schedule-executions/:executionId/complete-move`
3. 성공 시 `pending -> completed` 전이 + 결과 이벤트 연결(`result_ref_type/id`)

#### 3.8.3 예정 없이 바로 완료(현장 입력)

- 예정건이 없어도 "바로 완료" 허용
- 처리 원칙:
  1. 내부적으로 ad-hoc 실행건 생성(`pending`)
  2. 동일 트랜잭션에서 완료 처리
  3. 최종 상태 `completed`로 저장
- 목적: 현장 입력 편의성과 데이터 정합성 동시 보장

#### 3.8.4 검증/제약

- 권한: `farm_admin`, `farm_staff`만 등록/완료 허용
- 중복 완료 방지: `idempotency_key` 필수
- 상태 전이: 기본 `pending -> completed`
- 완료 실패 시: 도메인 반영/상태 전이 모두 롤백

### 3.9 농장 구조 설정 기반 초기입력 (운영 시작 전)

#### 3.9.1 진입 위치

- 초기 데이터 등록은 별도 페이지가 아니라 `/farm/admin`의 **농장 구조 설정** 내 단계로 제공한다.
- 구조(사/방/칸) 저장 후 동일 흐름에서 "초기입력"으로 이동한다.

#### 3.9.2 시설 타입별 입력 규칙

- 교배사/임신사/분만사:
  - 모돈 정보 입력 필수 (`sows`)
- 분만사:
  - 모돈 + 자돈/돈군 초기값 동시 입력 허용
- 기타 사육시설:
  - 돈군(`pig_groups`) + 두수 입력
- 초기입력의 돈군번호(`group_no`)는 서버 자동생성 정책을 기본으로 한다.

#### 3.9.2-a 교배사/임신사 수용 방식 설정 (스톨/군사)

- 설정 단위: 방(room)
- 저장 컬럼 권장: `farm_rooms.housing_mode`
  - `stall`: 스톨 방식(개체 중심)
  - `group`: 군사 방식(돈군/두수 중심)
- 기본값 권장: `group`

운영 규칙:
- `stall` 선택 시:
  - 초기입력 폼은 모돈 개체 배치 중심으로 노출
  - 검증 시 모돈 개체 수와 방/칸 재고 합계 정합성 체크
- `group` 선택 시:
  - 초기입력 폼은 돈군/두수 중심으로 노출
  - 모돈 정보는 정책에 따라 필수/선택 분기

UI 입력 방식(구조 설정 화면):
- 방(row)에 `운영방식` 버튼을 제공하고 클릭 시 통합 모달 오픈
- 모달 입력 항목:
  - 운영방식: `군사(group)` / `스톨(stall)`
  - 개수: 선택한 방식에 따라 라벨 동적 변경
    - 군사 선택: `칸 개수`
    - 스톨 선택: `스톨 개수`
- 저장 시 `housing_mode`와 하위 개수(섹션 수)를 함께 반영

#### 3.9.3 데이터 반영 규칙

- 모달 저장 시(섹션 단위) 단일 트랜잭션:
  1. `sows` 배치 반영 (`current_section_id`)
  2. `pig_groups` 생성 (`group_no` 자동생성, `birth_date` 저장)
  3. `section_inventory_ledger` opening IN 기록 (`ref_type='opening'`)
  4. `section_inventory_balance` upsert
  5. `pig_movement_events(event_type='entry')` + `pig_movement_lines(line_type='entry')` 기록
- 최종 확정 버튼은 두지 않고, 모달의 저장 버튼이 검증+DB 반영을 수행한다.
- 초기입력 완료 판정 전에는 분만/이동 완료 API를 제한한다.

전입일/이동일 반영 원칙(초기입력 연계):
- 전입일은 필수이며, opening 반영의 기준 시각으로 사용한다.
- 돈군 입력 시 `출생일` 또는 `일령` 중 1개 필수이며, `일령` 입력 시 출생일을 계산 저장한다.
- 전입일을 선택/변경하면 전입일 시점의 일령으로 출생일을 자동 역산한다.
- 출생일이 결정되면 일령은 오늘 기준(`today - birthDate`)으로 자동 재계산해 표시한다.
- 최소 반영 대상:
  - `pig_groups.current_section_id`
  - `pig_groups.birth_date`
  - `section_inventory_ledger.occurred_at`
- 출생일 저장 위치 기준(현재 구현):
  - **정본(돈군)**: `pig_groups.birth_date`
  - **정본(모돈 개체)**: `sows.birth_date`
  - **보조 스냅샷(구조 화면)**: `farm_sections.birthDate`
  - 일정 생성/판정 시에는 `pig_groups.birth_date`와 이벤트 시각(`occurred_at`, `moved_at`)을 우선 참조한다.
- 일정/이력 추적성 강화를 위해 초기입력을 `pig_movement_events(event_type='entry')`/`pig_movement_lines(line_type='entry')`로 병행 기록한다.
- 필요 시 `farm_sections.entryDate`도 동일 날짜로 동기화한다.

#### 3.9.4 API 권장

- `POST /api/farms/:farmId/bootstrap/opening/sections/:sectionId/save`
- `GET /api/farms/:farmId/bootstrap/opening/status`

구현 현황:
- `GET /api/farms/:farmId/bootstrap/opening/status` 구현 완료
  - 기준 컬럼: `farms.farm_initialized_at`
  - 응답: `initialized`, `farmInitializedAt`
- `POST /api/farms/:farmId/bootstrap/opening/sections/:sectionId/save` 구현 완료
  - 저장 시 해당 섹션의 입력 검증 + DB 반영을 한 트랜잭션으로 수행
  - `sows`, `pig_groups(birth_date 포함)`, `section_inventory_ledger(ref_type='opening')`, `section_inventory_balance` 반영
  - 전입일 입력 시 `occurred_at`/`moved_at`에 동일 반영
  - `pig_movement_events(event_type='entry')`, `pig_movement_lines(line_type='entry')` 반영
  - `schedule_executions` 완료건 자동 생성 반영
    - `execution_type='inspection'`, `status='completed'`
    - `result_ref_type='opening_section'`, `result_ref_id=section_id`
    - `scheduled_date=entryDate`, `completed_by=저장 사용자`
    - `work_plan_id`는 농장 단위 자동 work plan(`재고두수등록(초기값)`)을 재사용/생성
  - 응답 필드에 `scheduleExecutionId` 포함
  - 레거시 DB에서 `farm_sections.birthDate` 누락 시 `scripts/alter_farm_sections_add_entry_birth_date.sql` 적용 권장
  - 운영 반영 체크리스트: `docs/schedule_executions_rollout_checklist.md`

#### 3.9.5 분만 사전 조건

- 분만 완료 API 호출 전 대상 칸에 유효 모돈(`sows`) 존재 여부를 검증한다.
- 모돈이 없는 분만사 칸/빈 칸에서 분만 완료는 실패 처리한다.

#### 3.9.6 수용 방식 변경 가드

- 운영 시작 전(`farm_initialized_at` 미확정): `housing_mode` 변경 허용
- 운영 시작 후:
  - 기본 정책은 변경 제한
  - 예외 허용 조건: 대상 방의 활성 돈군/모돈/재고가 0
  - 변경 시 감사 로그(변경 전/후, 사용자, 시각, 사유) 기록 권장

## 4. 분만 확정 운영 흐름 (기준 정의: 출산일/분만일)

일정 마스터 페이지의 기준 정의 목록에 `출산일(분만일)` 항목이 있는 경우,  
해당 작업은 일정관리 페이지에서 "예정 -> 선택 -> 분만 등록 -> 확정" 흐름으로 처리한다.

### 4.1 마스터 설정 조건

- 기준 정의: `출산일(분만일)`
- 작업 내용(예): `분만 등록`
- 구조 위치: 분만사 해당 칸(또는 분만 대상 칸)

### 4.2 일정 생성/표시

- 마스터 규칙에 따라 `schedule_executions` 생성
  - `execution_type = 'birth'`
  - `status = 'pending'`
  - `scheduled_date = 예정 분만일`
- 일정관리 페이지는 `scheduled_date` 기준으로 해당 칸에 예정건 표시

### 4.3 작업자 실행(분만 등록)

- 작업자는 일정관리 페이지에서 예정건 클릭
- 분만 등록 모달 입력:
  - `bornCount` (분만 마리수)
  - `sectionId` (대상 칸)
  - `memo` (옵션)
  - `idempotencyKey` (중복 방지 키)
- API: `POST /api/farms/:farmId/schedule-executions/:executionId/complete-birth`

### 4.4 분만 확정 트랜잭션

1. `schedule_executions` 행 잠금 (`FOR UPDATE`)
2. 상태가 `pending`인지 확인
3. `farrowing_events` 생성
4. `pig_groups` 자동 생성
5. `section_inventory_ledger`에 IN 기록 (두수 증가)
6. `schedule_executions`를 `completed`로 변경
7. `result_ref_type='farrowing_event'`, `result_ref_id=<farrowing_events.id>` 저장
8. 필요 시 후속 일정(이동 예정) 자동 생성

### 4.5 예외/보완 규칙

- 예정건이 없는 경우:
  - "즉시 일정 추가 + 분만 확정" 허용
  - 내부적으로 ad-hoc execution 생성 후 동일 트랜잭션으로 완료 처리
- 중복 완료 방지:
  - 같은 execution은 1회 완료만 허용
  - 동일 idempotency key 재호출은 idempotent 처리
- 유효성:
  - `bornCount > 0`
  - `farmId`/권한 일치 검증
  - 동일 칸 동일 시점 중복 분만 방지 규칙 적용

## 5. 이동 확정 운영 흐름 (작업 유형: 이동)

일정 마스터 페이지의 작업 유형 목록에 `이동`이 설정된 일정은  
일정관리 페이지에서 "예정 -> 선택 -> 이동 입력 -> 확정" 흐름으로 처리한다.

### 5.1 마스터 설정 조건

- 작업 유형: `이동`
- 작업 내용(예): `자돈사 이동`, `합군 이동`, `분할 이동`
- 구조 위치: 출발 칸/도착 칸이 명확한 작업

### 5.2 일정 생성/표시

- 마스터 규칙에 따라 `schedule_executions` 생성
  - `execution_type = 'move'`
  - `status = 'pending'`
  - `scheduled_date = 예정 이동일`
- 일정관리 페이지는 **해당 날짜 + 해당 칸(section_id)** 셀에 이동 예정건을 표시
- 방(room) 행에는 이동 실행건을 직접 표시하지 않는다(칸 실행건 집계/요약만 허용)

### 5.3 작업자 실행(이동 등록)

- 작업자는 예정건 선택 후 이동 실행 모달 입력
- 입력 예시:
  - `eventType` (`full`, `partial`, `split`, `merge`)
  - `lines[]` (sourceGroupId, toSectionId, headCount, lineType)
  - `memo`, `idempotencyKey`
- API: `POST /api/farms/:farmId/schedule-executions/:executionId/complete-move`

### 5.4 이동 확정 트랜잭션

1. `schedule_executions` 행 잠금 (`FOR UPDATE`)
2. 상태가 `pending`인지 확인
3. `pig_movement_events` + `pig_movement_lines` 생성
4. 출발 칸 재고 OUT / 도착 칸 재고 IN (`section_inventory_ledger`)
5. `pig_groups` 두수/위치/상태 갱신
6. `schedule_executions`를 `completed`로 변경
7. `result_ref_type='movement_event'`, `result_ref_id=<pig_movement_events.id>` 저장

### 5.5 이동 정책 반영(확정값 연동)

- 전량 이동: 돈군 ID 유지, 위치만 변경
- 부분 이동/분할: 이동분은 신규 돈군 생성, 잔류는 source 유지
- 합군: 결과는 1돈군으로 정리
  - target은 **마리수 많은 돈군 우선**
  - 동수면 **created_at이 더 이른 돈군 우선**
  - source 두수 0이면 `merged`/`closed`

### 5.6 예외/보완 규칙

- 예정건이 없는 경우:
  - "즉시 일정 추가 + 이동 확정" 허용
  - 내부적으로 ad-hoc execution 생성 후 동일 트랜잭션 처리
- 중복 완료 방지:
  - 같은 execution 1회 완료 원칙
  - 동일 idempotency key 재호출은 idempotent 처리
- 유효성:
  - `headCount > 0`
  - 출발 돈군 두수 부족 시 실패
  - `farmId`/권한 일치 검증

## 6. `/farm/schedule` 개발 진행 순서 (계획서 기준)

아래 순서로 진행하면 `schedule_work_plans(계획) -> schedule_executions(예정/완료)` 구조를 안전하게 구현할 수 있다.

### 6.0 운영 초기입력 선행

1. 농장 구조 설정에서 칸 구조 완성
2. 교배사/임신사 방별 `housing_mode(stall/group)` 설정
3. 초기입력 단계에서 모돈/돈군 opening 데이터 등록
4. 검증/확정 완료 후 일정 완료 기능 오픈

### 6.1 백엔드 선행 고정

1. `schedule_executions` 테이블/인덱스/제약 생성
2. 상태 전이 규칙 고정: `pending -> completed`
3. 라우트 고정:
   - `POST /api/farms/:farmId/schedule-executions` (예정 등록)
   - `POST /api/farms/:farmId/schedule-executions/:executionId/complete-birth`
   - `POST /api/farms/:farmId/schedule-executions/:executionId/complete-move`

### 6.2 실행건 API 구현

1. 실행건 조회 API 구현 (기간/칸/상태 필터)
2. 캘린더 셀에서 예정 등록 API 구현 (`pending` 생성)
3. `idempotency_key` 검증/중복 처리 적용

### 6.3 완료 API 구현 (도메인 반영 포함)

1. 분만 완료 API:
   - `farrowing_events`, `pig_groups`, `section_inventory_ledger(IN)` 반영
   - `schedule_executions`를 `completed`로 전이
2. 이동 완료 API:
   - `pig_movement_events/lines`, `ledger(OUT/IN)`, `pig_groups` 반영
   - `schedule_executions`를 `completed`로 전이
3. `result_ref_type/result_ref_id` 연결 저장

### 6.4 프론트(`/farm/schedule`) 연동

1. 실행건 조회/예정 등록/완료 API 클라이언트 연결
2. 캘린더 셀에 `pending` 실행건 표시
3. 캘린더 셀 액션 구현:
   - 예정 등록 모달
   - 예정 선택 후 완료 모달
   - 예정 없이 "바로 완료" 모달(내부 pending 생성 후 completed 전이)
4. 완료 성공 후 셀 상태 즉시 갱신(낙관 갱신 또는 재조회 정책 고정)

### 6.5 검증/테스트

1. 상태 전이 테스트: `pending -> completed`
2. 중복 완료/idempotency 테스트
3. 원장 합계와 현재고 정합성 테스트
4. 권한/농장 격리(`farm_id`) 테스트

농장 선택은 **localStorage `currentFarmId`** 로만 결정된다. 값이 없으면 "농장을 선택한 후 이용해 주세요" 문구와 농장 선택 링크만 표시된다.

### 7.2 데이터 로딩

`farmId` 존재 시 다음 API를 병렬 호출한다.

| API | 용도 |
|-----|------|
| `getFarmFacilitiesTree(farmId)` | 건물→동(barn)→방(room)→칸(section) 트리 |
| `getFarmStructureProduction(farmId)` | 시설별 템플릿 ID·이름 매핑 (templateId, name) |
| `getFarmScheduleWorkPlansMaster(farmId)` | 해당 농장에 적용되는 기초 일정(work plan) 목록 |
| `getStructureTemplates()` | structure_templates 전역 목록 (테마색 등) |
| `getFarmSectionInventoryBalances(farmId)` | 칸별 재고 두수 (sectionId, headCount) |

추가로 기간 내 실행건 조회·동기화:

- `syncFarmScheduleExecutionsFromOpening(farmId)` — opening 기반 예정 동기화(멱등)
- `getFarmScheduleExecutions(farmId, { startDate, endDate, status: 'pending' | 'completed', limit })` — 주간 예정·완료 목록

주간 범위는 `weekStart`(해당 주 일요일) 기준 7일이며, `weekStart` 변경 시 실행건 목록을 다시 불러온다.

### 7.3 UI 구조

- **상단 툴바**
  - 현재 주 표시(월 레이블, 이전/다음 주·일, 캘린더 아이콘, 오늘 버튼)
  - 주간 날짜 범위 텍스트 (예: `2025-02-23 ~ 2025-03-01`)

- **메인 테이블**
  - **좌측 고정열**: "농장구조(재고두수)"
    - 계층: 건물(building) → 동(barn) → 방(room) → 칸(section)
    - 각 행 접기/펼치기(▸/▾), 단계별 일괄 펼치기/접기 버튼(▸/▾)
    - 동·방·칸 행에 재고 두수 표시
  - **우측 7열**: 요일별 날짜(일~토, 오늘 강조, 주말/공휴일 스타일)
  - **칸(section) × 날짜 셀**
    - 해당 칸·날짜의 **예정(pending)** 실행건: 태그(분만/이동/점검) + 작업명, 클릭 시 "작업 선택" 모달
    - **초기값 등록(opening) 완료** 실행건: "완료" 태그 + 작업명, 클릭 시 "재고두수등록(초기값) 완료 상세" 모달
    - **빈 영역**: 칸을 누르면 **예정 작업 입력** 모달이 바로 열림(작업 계획·실행 유형·메모 입력 후 등록). 예정/완료 항목이 있는 칸에서는 해당 항목을 눌러 작업 선택·상세로 이동

- **칸별 후보 work plan**
  - 각 section 행에는 해당 동(barn)의 `structure_template_id`(또는 barnType)에 매핑된 `schedule_work_plans` 목록이 `candidateWorkPlanIds`로 넘어가며, 빈 칸 "+" 또는 바로 완료 시 이 목록으로 작업 계획을 선택한다.

### 7.4 모달·플로우

| 모달 | 트리거 | 동작 |
|------|--------|------|
| **캘린더** | 툴바 캘린더 아이콘 | 월별 날짜 선택 → 선택일로 `selectedDateStr`·`weekStart` 변경 |
| **예정 작업 입력** | 캘린더에서 칸(칸×날짜 셀) 클릭 | 칸·날짜가 고정된 상태로 바로 열림. 작업 계획(work plan) 선택, 실행 유형(birth/move/inspection), 메모 입력 후 등록 → `createFarmScheduleExecution` 호출, 셀에 예정건 반영. (해당 칸에 예정/완료 항목이 있으면 항목 클릭 시 작업 선택·상세 모달로 이동) |
| **바로 완료** | (현재는 예정건 완료 등 다른 경로) | 작업 계획·실행 유형·작업일·칸 등 입력 → direct complete API 호출(분만/이동) |
| **작업 선택** | 예정건 클릭 | "완료 처리" / "수정" 버튼 → 완료 처리 시 완료 모달, 수정은 "예정 수정(준비 중)" |
| **완료 처리 / 바로 완료** | 예정건 완료 또는 바로 완료 | 유형별 입력: **분만** — 마리수, sectionId, groupNo, originSowId, 메모; **이동** — 이벤트 유형(전량/부분/분할/합군/출하), 도착 칸, 마리수, 메모. **시설/점검** — 완료 API 미연결 안내 |
| **재고두수등록(초기값) 완료 상세** | opening 완료건 클릭 | 작업명, 전입일, 두수, 돈군번호, 출생일, 일령, 모돈수, 완료시각, 메모 등 읽기 전용 |
| **예정 수정** | 작업 선택 → 수정 | 현재 "수정 기능은 준비 중입니다" 안내만 표시 |

### 7.6 상태·정책 요약

- **표시 단위**: 실행건은 **칸(section) + scheduled_date** 단위로만 셀에 표시된다. 방(room) 행에는 실행건을 직접 넣지 않고, 방/동 두수 등 구조 요약만 표시한다.
- **실행 유형**: `birth`(분만), `move`(이동), `inspection`(시설/점검). 작업 계획 라벨에서 키워드로 유형 추론(`inferExecutionTypeFromPlan`).
- **시설 전용 계획**: `isFacilityOnlyPlan()`으로 "시설/점검" 성격이면 빈 칸(두수 0)에서도 해당 계획을 후보로 노출할 수 있다.
- **비육 도착 칸**: 도착 칸 옵션에서 비육사일 경우 "출하" 이벤트 유형을 선택할 수 있다.
