# DB 구조 조언 — 단일 DB vs 분리

현재 프로젝트는 **하나의 PostgreSQL DB**에 회원·관리자 설정·농장·일정 등 모든 테이블이 들어가는 구조입니다.  
“회원은 별도 DB, admin 정보는 별도 DB, 농장은 현재 DB”처럼 나누는 게 맞는지에 대한 조언입니다.

---

## 1. 현재 구조 요약

| 구분 | 테이블 예시 | 관리 화면 | 비고 |
|------|-------------|-----------|------|
| 회원·권한 | users, roles, user_farms | admin.html (사용자/역할) | 시스템 전역 |
| 관리자 설정 | structure_templates, schedule_*, disease_codes, vaccine_types, feed_types, breeds | admin.html (일정·시설 기준·코드 등) | 전역 마스터 |
| 농장·농장별 데이터 | farms, farm_*, farm_schedule_*, pig_groups, pigs, pig_movements | farm_admin.html, 대시보드 | farmId로 구분 |

→ **한 DB 안에** “시스템/전역” 테이블과 “농장별” 테이블이 함께 있는 구조입니다.

---

## 2. 일반적인 패턴 세 가지

### (1) 단일 DB, 테이블로 구분 (현재 방식)

- **구조**: DB 1개, 테이블 이름/접두어로 역할 구분 (users, schedule_*, farm_* 등).
- **장점**: 배포·백업·마이그레이션 단순, 트랜잭션·JOIN 자유로움, 운영 부담 적음.
- **단점**: 논리적 영역(회원 vs 관리 vs 농장)이 물리적으로 한곳에 있어, “완전 격리”는 아님.
- **쓰는 경우**: 중소 규모, 단일 서비스, 팀/운영 인력이 많지 않을 때. **가장 흔함.**

### (2) DB를 역할별로 분리 (회원 DB / 관리 DB / 농장 DB)

- **구조**: 예) `auth_db`(회원·역할), `admin_master_db`(일정 마스터·코드 등), `farm_db`(농장·일정 실행·돼지).
- **장점**: 영역별 백업·권한·규제 분리, 서로 다른 서버로 이전하기 쉬움.
- **단점**:  
  - 트랜잭션을 DB 경계를 넘어 쓰기 어렵고,  
  - JOIN 불가 → 애플리케이션에서 여러 DB 조회 후 합치거나, 복제/동기화 필요.  
  - 배포·마이그레이션·모니터링가 DB 개수만큼 늘어남.
- **쓰는 경우**:  
  - 회원 DB를 법·규정상 완전 분리해야 하거나,  
  - 팀/서비스가 “인증팀 / 관리팀 / 농장서비스”처럼 완전히 나뉘어 있을 때.

### (3) 한 DB 안에서 스키마만 분리 (PostgreSQL schema)

- **구조**: DB 1개, 스키마만 나눔. 예) `auth.users`, `admin.structure_templates`, `farm.farms`, `farm.farm_schedule_items`.
- **장점**:  
  - 논리적 구분(회원 / 관리 / 농장)이 명확해지고,  
  - 필요 시 스키마 단위 백업·권한 부여 가능.  
  - 같은 DB라서 필요하면 스키마를 넘는 JOIN·트랜잭션도 가능.
- **단점**: 초기 설계·마이그레이션 시 스키마 설계와 코드 수정 필요.
- **쓰는 경우**: “지금은 한 DB로 가되, 나중에 분리 가능성”을 열어두고 싶을 때.

---

## 3. 이 프로젝트에 대한 조언

### 결론: **지금 단계에서는 “하나의 DB + 테이블 구분”이 맞는 구조입니다.**

이유 요약:

1. **관계가 밀접함**  
   - 농장(farms) ↔ 사용자(user_farms, users), 일정 마스터(schedule_*) ↔ 농장 일정(farm_schedule_*) 등 **JOIN과 트랜잭션이 자주 필요**합니다.  
   - DB를 나누면 이걸 한 트랜잭션으로 묶기 어렵고, 설계·구현이 크게 복잡해집니다.

2. **운영 부담**  
   - DB를 2~3개로 나누면 백업·마이그레이션·모니터링이 배로 늘고,  
   - “회원 DB만 복구” 같은 요구가 없으면 이 비용 대비 이득이 크지 않습니다.

3. **일반적인 관례**  
   - 중소형 비즈니스/관리 시스템은 **한 DB에 여러 도메인 테이블**을 두고,  
   - 테이블 이름·폴더(모델/라우트)로 “회원 / 관리 / 농장”을 구분하는 방식이 **가장 흔**합니다.

그래서  
- **“회원은 별도 DB, admin은 별도 DB, 농장만 현재 DB”** 처럼 **지금 당장 나누는 것은 권하지 않습니다.**  
- 대신 **현재처럼 하나의 DB를 쓰고**, 아래처럼 “논리적 구분”만 명확히 하는 쪽을 추천합니다.

---

## 4. 권장: 한 DB 유지 + 논리적 구분 강화

DB는 그대로 두고, **역할별로 테이블/코드만 명확히 구분**하는 방식입니다.

| 영역 | 테이블(예) | 관리 주체 | 비고 |
|------|------------|-----------|------|
| 회원·권한 | users, roles, user_farms | admin (시스템 관리자) | 이미 접두어/이름으로 구분 가능 |
| 전역 마스터(관리자 설정) | structure_templates, schedule_*, disease_codes, vaccine_types, feed_types, breeds | admin.html | “전역” 주석/문서로 표시 |
| 농장·농장별 업무 | farms, farm_*, farm_schedule_*, pig_*, section_group_occupancy | farm_admin / 대시보드 | farmId 필수 |

추가로 할 수 있는 것:

- **백업**: 같은 DB라도 스크립트에서 “회원 관련 테이블만 덤프” / “농장 관련만 덤프” 식으로 **논리별 백업**은 가능합니다.
- **권한**: DB 사용자(role)를 나눌 필요가 생기면, 나중에 **PostgreSQL 스키마**를 도입해 `auth`, `admin`, `farm` 으로 옮기는 식으로 단계적 분리를 검토할 수 있습니다.

---

## 5. 정리

| 질문 | 답변 |
|------|------|
| 지금처럼 하나의 DB에 다 두는 구조가 일반적인가? | **예.** 중소형 시스템에서 가장 흔한 방식입니다. |
| 회원·admin·농장을 DB 단위로 나눠야 하나? | **지금 단계에서는 아니다.** JOIN·트랜잭션·운영이 복잡해져서 비추입니다. |
| 나중에 “회원만 다른 DB로” 분리하고 싶다면? | 그때는 (3)처럼 **먼저 한 DB 안에서 스키마 분리(auth, admin, farm)**를 검토하고, 정말 필요할 때만 DB 인스턴스를 나누는 편이 안전합니다. |

**요약:**  
- **현재처럼 “하나의 DB + 테이블로 역할 구분”이 일반적이고, 이 프로젝트 규모/구조에도 맞는 선택입니다.**  
- admin에서 쓰는 정보(일정 마스터·시설 기준 등)와 농장에서 쓰는 정보(농장별 일정·돼지 등)를 **같은 DB에 두고 farmId·테이블 이름으로 구분하는 현재 방식**을 유지하고,  
- 필요하다면 “논리적 영역(회원 / 관리 / 농장)”만 문서·백업·코드 구조로 명확히 하는 것을 권합니다.
