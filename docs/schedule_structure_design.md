# 돈사 작업 일정 스케줄 — 조건별 구조 설계

기존 개념을 두지 않고, **구분·대상장소·기준·작업유형** 4가지 축으로 돈사 작업 일정을 설계한 문서입니다.

---

## 1. 네 가지 축(구조) 정의

| 축 | 한글명 | 설명 | 예시 |
|----|--------|------|------|
| **구분** | 사육/대상 구분 | 어떤 개체·대상을 기준으로 할지 | 모돈, 옹돈, **자돈**, 비번식돈, 시설 |
| **대상장소** | 대상 장소(시설) | 어느 돈사/시설을 대상으로 할지. **기존 `structure_templates` 테이블과 연동** (교배사, 임신사, 분만사 등은 템플릿의 name으로 등록) |
| **기준** | 기준일/기준 사건 | “언제부터”를 셀 기준. **구분=시설이어도 돼지 전입일·입식일 등 사용 가능** (해당 장소에 돼지가 들어온 날 기준). | 전입일, 입식일, 교배일, 유산일, 임신확정, 발정일, … |
| **작업유형** | 작업 종류 | 어떤 종류의 작업인지 | 이동, 사양, 질병, 환경, … |

- **자돈**: 분만사에서 분만 시 태어난 새끼(분만 후 ~ 이유 전). 비번식돈은 “이유 후 자돈사·육성사·비육사”에 있는 개체 구분이고, **분만사에 있는 새끼는 구분을 “자돈”**으로 둡니다.

- **일정 항목** = (구분, 대상장소, 기준, 작업유형) + “기준일로부터 N일~M일” 등으로 정의.
- **구분**에 따라 **적용 가능한 대상장소**가 정해짐 → “모돈 → 교배사, 임신사, 분만사, 격리사, 후보사”처럼 테이블로 정리.

## 2. 일정(작업) 추가 시 선택 흐름

### 2.1 선택 순서 요약

```
① 장소 → ② 구분 → ③ 기준 유형 → ④ 작업유형(대분류 → 세부) → ⑤ 일수·설명
```

### 2.2 단계별 선택 흐름

| 단계 | 선택 항목 | 노출/선택 내용 | 비고 |
|------|-----------|----------------|------|
| **① 장소** | 대상장소 | **structure_templates** 전체 목록. **전체 시설 반복**일 경우 “전체 시설” 선택 또는 장소 생략(appliesToAllStructures=true). | “어디서 할 작업인가?” |
| **② 구분** | 구분 | **①에서 고른 장소**에 매핑된 구분만 노출 (schedule_division_structures 조회). 예: 분만사 선택 시 → 모돈, 자돈, 시설 | “이 장소에서 모돈? 자돈? 시설?” |
| **③ 기준 유형** | 기준 | **②에서 고른 구분**에 맞는 기준만 노출 (전입일, 입식일, 교배일, 분만일, 임신확정 등). 시설 선택 시에도 전입일·입식일 사용 가능 | “언제부터 셀 기준일?” |
| **④ 작업유형** | 대분류 → 세부 | 1) **대분류** 셀렉트 선택 (W01 이동, W02 사양, W05 환경 등). **② 구분**에 따라 노출 범위 제한. 2) **세부** 셀렉트 선택 (분만사 이동, 온도 조절, 교배 실시 등). 대분류 선택 시 해당 대분류에 속한 세부만 옵션으로 노출. **둘 다 셀렉트(드롭다운) 선택.** | “무슨 종류의 작업?” |
| **⑤ 일수·설명** | dayMin, dayMax, description | 기준일로부터 **시작 일수**(dayMin), **끝 일수**(dayMax). 필요 시 작업 내용(description) 입력 | “기준일 N~M일 후” / “0~7일 구간” 등 |

### 2.3 흐름 요약

- **① 장소**로 “어디서”를 정한 뒤 → **② 구분**으로 “그 장소에서 누구(모돈/자돈/시설 등) 대상” → **③ 기준**으로 “무슨 날짜/사건 기준으로 셀지” → **④ 작업유형**으로 “무슨 작업(대분류→세부)” → **⑤ 일수·설명**으로 “기준일로부터 며칠 구간, 설명” 순서로 진행하면 됩니다.
- ② 구분은 “선택한 장소에 적용 가능한 구분”만, ④ 작업유형은 “선택한 구분에 시설 해당/개체 해당”에 맞는 것만 노출하면 됩니다.
- **④ 작업유형 UI**: 대분류·세부 **둘 다 셀렉트(드롭다운) 선택**. 대분류 셀렉트에서 항목 선택 후 → 세부 셀렉트에 해당 대분류의 세부 항목만 옵션으로 채워서 선택.

### 2.4 대분류·세부가 목록에 없을 때

작업 추가 시 필요한 **대분류**나 **세부**가 셀렉트 목록에 없는 경우, 아래 둘 중 하나(또는 병행)로 처리할 수 있습니다.

| 방식 | 설명 | 장점 |
|------|------|------|
| **A. 별도 관리 화면** | admin 등에 **「작업유형 대분류 관리」**, **「세부 작업유형 관리」** 메뉴를 두고, 여기서 대분류(W01~W10)·세부를 먼저 등록. 작업 추가 시에는 등록된 목록만 셀렉트에 노출. 목록에 없으면 “먼저 관리 화면에서 추가 후 다시 작업 추가” 안내. | 마스터가 한곳에서 정리됨. 코드·순서·시설 해당 여부 등을 체계적으로 관리 가능. |
| **B. 작업 추가 폼에서 인라인 추가** | 대분류/세부 셀렉트 **하단에 「추가」 버튼** 배치. 클릭 시 모달에서 현재 목록 확인 + 하단에서 신규 입력 후 저장. (아래 B-1 UI 참고) | 작업 추가 중에 바로 새 대분류/세부를 만들어 쓸 수 있어 편리. |

**B-1. 셀렉트 하단 추가 버튼 + 모달 UI (권장)**

- **작업 추가 폼**: 대분류 셀렉트·세부 셀렉트 **아래에 각각 「추가」 버튼** 배치.
- **추가 버튼 클릭 시** → **모달** 오픈.
  - **상단**: **현재 목록** 표시 (대분류 추가 모달이면 schedule_work_types 목록, 세부 추가 모달이면 선택된 대분류에 속한 schedule_work_detail_types 목록을 테이블/리스트로 표시).
  - **하단**: **신규 입력 영역** (이름, 필요 시 코드·appliesToScope 등). 저장 버튼 클릭 시 API로 INSERT 후 모달 닫고, 셀렉트 목록 갱신 + 방금 추가한 항목 자동 선택.
- **효과**: 기존 항목을 보면서 “없는 것만 추가”하기 쉽고, 같은 화면에서 목록 확인과 입력을 한 번에 처리할 수 있음.

- **권장**: 기본은 **A(관리 화면)** 로 대분류·세부를 등록해 두고, **B-1(셀렉트 하단 추가 버튼 + 모달)** 을 세부만 허용(대분류는 고정, 세부만 모달로 추가)하는 식으로 함께 써도 됩니다.
- **저장 위치**: 신규 대분류 → `schedule_work_types` INSERT, 신규 세부 → `schedule_work_detail_types` INSERT (workTypeId 지정). 저장 후 해당 행이 셀렉트 옵션에 포함되도록 목록 API/캐시를 갱신하면 됩니다.

**기준 유형이 목록에 없을 때**  
기준 유형도 **작업 추가 폼에서만** 처리합니다. 기준 셀렉트 **하단에 「추가」 버튼**을 두고, 클릭 시 모달(상단: 현재 기준 목록, 하단: 신규 입력)에서 추가 후 저장 → 셀렉트 갱신·방금 추가한 항목 자동 선택. 별도 “기준 유형 관리” 화면은 두지 않습니다.

**장소·구조**  
**장소(대상장소)** 와 시설 구조는 **농장 구조 설정**에서만 등록·관리합니다. `structure_templates`는 농장 구조 템플릿이므로, 일정 폼에서는 이미 등록된 목록만 셀렉트로 노출합니다. 구분(divisions) 마스터는 전역 5개(모돈·옹돈·자돈·비번식돈·시설) 수준이면 초기 데이터로 두고, 추가는 관리 화면 또는 초기 스크립트로만 할 수 있음.

## 3. 구분별 적용 대상장소 (구조 정리)

아래는 “구분 → 이 구분에 적용하는 대상장소”를 정리한 예시입니다.  
대상장소는 **기존 `structure_templates`** 테이블의 행(교배사, 임신사, 분만사 등)을 사용하고, 실제 “구분별 적용 장소” 데이터는 **구분–대상장소 매핑 테이블**에서 `structure_templates.id`를 FK로 저장합니다.

| 구분 | 적용 대상장소 (예시) |
|------|----------------------|
| **모돈** | 교배사, 임신사, 분만사, 격리사, 후보사 |
| **옹돈** | 웅돈사, 격리사 (필요 시 추가) |
| **자돈** | **분만사**, 격리사 (필요 시) |
| **비번식돈** | 자돈사, 육성사, 비육사, 격리사 (필요 시 추가) |
| **시설** | 교배사, 임신사, 분만사, 자돈사, 육성사, 비육사, 격리사, 후보사, 웅돈사 (시설별 반복 작업 등) |

- **모돈**: 교배 → 임신 → 분만 흐름. 교배사·임신사·분만사·격리사·후보사.
- **옹돈**: 웅돈사 중심, 격리사.
- **자돈**: 분만사에서 분만 시 새끼(분만 후 ~ 이유 전). **분만 시 새끼가 있는 경우 구분 = 자돈, 대상장소 = 분만사.** 이유 후 자돈사로 이동하면 비번식돈(또는 자돈사 구간만 별도 관리)으로 볼 수 있음.
- **비번식돈**: 이유 후 자돈사·육성사·비육사 흐름. 자돈사·육성사·비육사·격리사.
- **시설**: 모든 장소에서 “시설 기준” 반복 작업(환경·청소 등) 가능.

## 4. 테이블 구조 제안

### 4.1 마스터 테이블 (4개 축)

#### (1) schedule_divisions — 구분

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL PK | 고유 ID |
| code | VARCHAR(50) | 코드 (sow, boar, piglet, non_breeding, facility) |
| name | VARCHAR(100) | 표시명 (모돈, 옹돈, 자돈, 비번식돈, 시설) |
| sortOrder | INTEGER | 정렬 순서 |

#### (2) 대상장소 — 기존 `structure_templates` 연동

대상장소(교배사, 임신사, 분만사, 자돈사, 육성사, 비육사, 격리사, 후보사, 웅돈사 등)는 **별도 테이블을 만들지 않고**, 이미 있는 **`structure_templates`** 테이블을 사용합니다.

| 항목 | 설명 |
|------|------|
| **테이블** | `structure_templates` (기존) |
| **모델** | `StructureTemplate` (models/StructureTemplate.js) |
| **대상장소로 사용** | 각 행의 `id`, `name` (예: 분만사, 자돈사), `sortOrder` 등 |
| **연동 방식** | 일정 쪽에서는 `structureTemplateId`(FK → structure_templates.id)로 참조. 구분별 적용 장소는 매핑 테이블에서 `structureTemplateId`로 저장. |

- 농장 구조 템플릿에 등록된 시설(교배사, 임신사, 분만사 등)이 그대로 “일정의 대상장소” 목록이 됨.
- 새 시설 유형을 추가할 때는 `structure_templates`에만 등록하면 되고, 일정용 별도 마스터는 관리하지 않음.

#### (3) schedule_bases — 기준

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL PK | 고유 ID |
| code | VARCHAR(50) | 코드 |
| name | VARCHAR(100) | 표시명 (전입일, 입식일, 교배일, …) |
| divisionId | INTEGER FK nullable | 주로 쓰는 구분 (NULL이면 전 구분 공통). **시설 구분에서도 전입일·입식일 등 선택 가능**하도록 NULL 또는 시설 포함 가능. |
| sortOrder | INTEGER | 정렬 순서 |

**예시**

| code | name | 비고 |
|------|------|------|
| entry_date | 전입일 | 개체·**시설 공통** (시설일 때 = “해당 장소로 돼지 전입한 날”) |
| placement_date | 입식일 | 개체·시설 공통 |
| breeding_date | 교배일 | 모돈 |
| abortion_date | 유산일 | 모돈 |
| pregnancy_confirmed | 임신확정 | 모돈 |
| estrus_date | 발정일 | 모돈 |
| farrowing_date | 분만일 | 모돈 |
| weaning_date | 이유일 | 모돈/자돈 |
| … | … | 필요 시 추가 |

**시설 + 돼지 기준일 조합**  
구분이 **시설**이어도, **기준**은 “돼지/돈군이 그 장소에 전입·입식한 날”을 쓸 수 있습니다.  
- 예: **시설**, 장소 **분만사**, 작업유형 **환경(온도 조절)**, 기준 **전입일**, dayMin=1, dayMax=1  
  → “분만사에 돼지 전입일 기준 1일 후에 온도 조절(환경) 작업”을 수행하는 일정.  
- 즉, **무슨 작업인지(시설 작업)**와 **언제 할지(돼지 전입일 기준)**는 별개로 두고, 시설 일정도 전입일·입식일 기준으로 스케줄링 가능합니다.

#### (4) 작업유형 — 대분류 + 세부유형 (2단계)

**제안 평가**  
다른 설계에서 제안한 **작업유형 대분류(W01~W10) + 세부 작업유형(WD001~)** 구조를 채택합니다.  
- **장점**: 실무 용어·흐름과 맞고, W01/W02 코드로 필터·통계 시 사용하기 좋음. 기록(W08)·도태(W09)·시설(W10)·위생(W06)·점검(W07)이 따로 있어서 일정·완료 처리 설계에 잘 맞음.  
- **구조**: 대분류 1건 → 세부유형 N건. 일정 항목은 **세부유형**을 FK로 두고, 대분류는 조인으로 조회.

**정의**  
- **대분류(work_type)**: 이동·사양·번식·질병·환경·위생·점검·기록·도태·시설 등 10개. 필터·그룹·통계용. DB/시드에 code(W01~W10)가 있을 수 있으나 **화면 표시는 이름만** 사용.  
- **세부유형(work_detail_type)**: 대분류 아래 실제 작업명(분만사 이동, 교배 실시, 예방접종 등). 일정 항목이 참조하는 것은 **세부유형**입니다.

**작업유형 = 행동 기준, 상태와 작업 구분**  
작업유형은 **“무엇을 할지”(행동·작업)** 로만 정의하고, **“어떤 상태인지”** 는 작업유형에 넣지 않습니다.

| 구분 | 의미 | 예시 | 작업유형으로 둘 것 |
|------|------|------|---------------------|
| **상태** | 상황·조건 (무엇인지) | 임신중, 분만대기, 격리중, **분만 예정** | **X** — 작업유형에 등록하지 않음. 기준(기준일)이나 별도 상태 값으로 사용. |
| **작업** | 수행할 행동 (무엇을 할지) | 임신확정 **검사**, **분만사 이동**, **분만 준비** | **O** — 작업유형(세부)으로 등록. “~하기”, “~실시”, “~이동”, “~검사” 등 행동 단위. |

- **잘못된 예**: “분만 예정”을 세부 작업유형으로 등록 → 상태를 작업처럼 둔 것.  
- **올바른 예**: “분만 준비”(분만 전에 할 작업), “분만사 이동”, “임신확정 검사” → 실제로 수행하는 행동.  
- **상태**는 **기준(basis)** 쪽에 “분만 예정일”, “임신확정일”처럼 **기준일/사건**으로 두거나, 돈군·개체의 “현재 상태”를 다루는 별도 필드/마스터로 분리합니다. 일정 항목은 “그 상태/기준일을 기준으로 **무슨 작업을 할지**”만 작업유형에 넣습니다.

**시설 해당 여부**  
작업유형에는 **구분 = 시설**일 때만 쓰는 것과 **구분 = 모돈/옹돈/자돈/비번식돈(개체)**일 때만 쓰는 것이 있고, 둘 다 쓰는 것도 있습니다.  
- **시설 해당**: 환경(W05), 위생(W06), 시설(W10) → 장소/시설 단위 작업(청소, 소독, 장비 점검 등). **구분 = 시설**일 때 선택 가능.  
- **개체 해당**: 이동(W01), 사양(W02), 번식(W03), 질병(W04) 등 → 돼지/돈군 단위. **구분 = 모돈/옹돈/자돈/비번식돈**일 때 선택 가능.  
- **둘 다**: 점검(W07), 기록(W08), 도태(W09) → 개체·시설 모두에서 사용 가능한 경우.  

일정 추가 시 **구분**을 선택한 뒤에는, 위 규칙에 맞는 작업유형(대분류·세부)만 노출하도록 하면 됩니다.

### 4.2 구분–대상장소 매핑 (조건별 구조)

**구분에 따라 “어떤 대상장소(structure_templates)를 선택할 수 있는지”** 를 저장합니다.

#### schedule_division_structures — 구분 ↔ 대상장소 (structure_templates 연동)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL PK | 고유 ID |
| divisionId | INTEGER FK | 구분 (schedule_divisions.id) |
| structureTemplateId | INTEGER FK | 대상장소 (**structure_templates.id**) |
| sortOrder | INTEGER | 표시 순서 (선택) |

- 동일 (divisionId, structureTemplateId) 중복 방지: UNIQUE(divisionId, structureTemplateId).
- 조회: “모돈” 선택 시 → divisionId = 모돈 → 이 테이블에서 structureTemplateId 목록 조회 → structure_templates와 조인해 교배사, 임신사, 분만사, 격리사, 후보사 등 이름 표시.

### 4.3 일정 항목 (실제 스케줄 정의)

#### schedule_items — 일정 항목

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | SERIAL PK | 고유 ID |
| divisionId | INTEGER FK | 구분 |
| structureTemplateId | INTEGER FK nullable | 대상장소 (**structure_templates.id**. division에 대해 매핑된 것만 허용 권장) |
| basisId | INTEGER FK | 기준 |
| workDetailTypeId | INTEGER FK | 작업유형 **세부** (schedule_work_detail_types.id). 대분류는 조인으로 조회. |
| dayMin | INTEGER | 기준일로부터 시작 일수 (예: 0) |
| dayMax | INTEGER | 기준일로부터 끝 일수 (예: 7) |
| description | TEXT | 작업 내용 설명 (선택) |
| sortOrder | INTEGER | 정렬 |
| isActive | BOOLEAN | 사용 여부 |
| **appliesToAllStructures** | BOOLEAN | **전체 시설 적용** 여부. true면 대상장소 무관 전체 시설에 반복 일정 적용(아래 반복 컬럼 사용). false 또는 null이면 structureTemplateId 지정 시 해당 장소만. |
| **recurrenceType** | VARCHAR(20) nullable | 반복 구분: **'none' \| 'daily' \| 'weekly' \| 'monthly'**. 전체 시설 반복 일정 시 사용. none = 비반복(기준일+dayMin~dayMax만). |
| recurrenceInterval | INTEGER nullable | 반복 간격 (예: 2주마다=2). 기본 1. |
| **recurrenceWeekdays** | VARCHAR(30) nullable | **주 단위** 시 요일. 0=일, 1=월, … 6=토. 복수 시 콤마 구분 (예: "1,3,5" = 월·수·금). |
| **recurrenceMonthDay** | INTEGER nullable | **월 단위** 시 일(1~31). 매월 해당 일에 실행. |
| recurrenceStartDate | DATE nullable | 반복 시작일. |
| recurrenceEndDate | DATE nullable | 반복 종료일. null이면 무기한. |
| createdAt, updatedAt | TIMESTAMP | 생성/수정 시각 |

- **대상장소**: 기존 `structure_templates`와 연동하므로 `structureTemplateId`로 저장. (분만사, 자돈사 등은 structure_templates에 등록된 행의 id.)
- **전체 시설 반복 일정**: **appliesToAllStructures = true** 이고 **recurrenceType = daily \| weekly \| monthly** 이면, “매일”/“주 단위(요일 지정)”/“월 단위(일 지정)”로 **모든 시설**에 적용되는 반복 일정. 장소는 특정하지 않음(structureTemplateId null 가능). 예: 매일 전체 시설 소독, 매주 월·목 전체 시설 점검, 매월 1일 전체 시설 환경 점검.
- 의미: “구분 = 모돈, 대상장소 = 분만사, 기준 = 전입일, 작업유형 세부 = 분만사 이동”이고 “전입일 0~7일”이면 → 분만사 전입일 0~7일 구간에 분만사 이동 작업을 수행하는 일정.

## 5. 구조 요약 (ER 관점)

```
structure_templates (대상장소, 기존 테이블)
       ▲
       │ structureTemplateId
       │
schedule_divisions (구분)
       │
       ├── schedule_division_structures ──► structure_templates (대상장소 연동)
       │
       └── schedule_items ─┬─ schedule_divisions (구분)
                           ├─ structure_templates (대상장소, structureTemplateId)
                           ├─ schedule_bases (기준)
                           └─ schedule_work_detail_types (작업유형 세부) ──► schedule_work_types (대분류)
```

- **대상장소**는 신규 테이블 없이 **structure_templates**와 연동. 구분별 적용 장소는 **schedule_division_structures**에서 (divisionId, structureTemplateId)로 저장.
- **구분** 1:N **구분–대상장소 매핑** N:1 **structure_templates** → “모돈 → 교배사, 임신사, 분만사, 격리사, 후보사” 형태로 정리.
- **일정 항목**은 구분·structureTemplateId·기준·작업유형을 FK로 참조하고, dayMin/dayMax(또는 반복)로 “언제”를 정의.

### 5.1 전역 vs 농장 (테이블 분리)

**전역(admin)과 농장(farm_admin)은 테이블을 따로 둡니다.**

| 구분 | 용도 | 테이블 예 |
|------|------|------------|
| **전역** | admin 일정 관리 설정. 템플릿·마스터. | schedule_divisions, schedule_division_structures, schedule_bases, schedule_work_types, schedule_work_detail_types, **schedule_items** (전역 일정 항목) |
| **농장 전용** | farm_admin 농장 일정 관리. 농장별 데이터. | **farm_schedule_*** (농장별 구분·장소매핑·기준·작업유형·일정 항목 등). 전역 테이블을 참조(originalId)하거나 복사해 두고, 농장별로 수정·추가한 내용 저장 |

- 전역에서 정의한 일정 항목(schedule_items)을 농장에 “복사”해 farm_schedule_items로 쓰는지, 농장에서 전역을 참조만 하는지는 구현 시 정하면 됩니다. **테이블 자체는 전역용과 농장용을 분리**합니다.
- 장소(structure_templates)는 농장 구조에서 공통으로 쓰고, 구분·기준·작업유형 마스터는 전역에 두고 농장은 전역 ID를 참조하는 구조를 많이 씁니다.

   - schedule_division_structures, schedule_items 등 신규 테이블에서 **structureTemplateId FK → structure_templates(id)** 로 연동. 기존 structure_templates 테이블/모델은 그대로 사용.
4. **UI**  
   - admin/farm_admin에서 “구분 선택 → 대상장소는 해당 구분에 맞는 structure_templates 목록만 노출”(schedule_division_structures로 필터)하도록 연동.

## 7. 보완·추가 사항 점검

아래는 현재 문서에 **함축만 되어 있거나 아직 안 쓴 항목**입니다. 구현 단계에서 필요하면 이 문서에 섹션을 추가하면 됩니다.

| 항목 | 반영 내용 |
|------|------------|
| **기준 유형이 목록에 없을 때** | **작업 추가 폼에서 추가 버튼으로만** 처리. 기준 셀렉트 하단 「추가」 버튼 → 모달(현재 목록 + 하단 입력)에서 신규 등록. 별도 관리 화면 없음. (§2.4 근처) |
| **장소·구조** | **농장 구조**에서만 등록·관리. structure_templates는 농장 구조 설정용. 일정 폼에서는 등록된 장소만 셀렉트 노출. (§2.4 근처) |
| **전역 vs 농장** | **전역 테이블(schedule_*)** 과 **농장 전용 테이블(farm_schedule_*)** 을 **따로 둠**. 전역=admin 템플릿, 농장=farm_admin 농장별 데이터. (§5.1) |
| **전체 시설 반복 일정** | **매일(daily)·주 단위(weekly)·월 단위(monthly)** 반영. schedule_items에 appliesToAllStructures, recurrenceType, recurrenceWeekdays, recurrenceMonthDay 등 컬럼 정의. (§4.3) |
| **일정 항목 목록·조회·수정·삭제** | (보완 시) 목록 조회·필터·수정 모달·삭제·정렬 변경 등 흐름 요약. |
